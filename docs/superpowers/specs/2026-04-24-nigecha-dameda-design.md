# nigecha-dameda（逃げちゃダメだ）設計ドキュメント

> 逃げちゃダメだ、逃げちゃダメだ、逃げちゃダメだ... — 碇シンジ

## 概要

Claude が「後続チケットで対応します」と言って今のPRで直せるものを先送りにする行動（サボタージュ）を防ぐ **Stop Hook**。

毎ターン自動発動し、先送りがあった場合にチケット番号の明記 or ユーザーへの確認を強制する。

---

## 問題

- 「後続チケットで対応します」と言ってスコープ内の修正を回避する
- 先送りにする場合でも、どのチケットで対応するかを言わない
- チケットが存在しないのに「後続チケット」と言って実質的に放置になる

正当な先送り（スコープ外・設計変更が必要・ユーザー合意あり）は存在するため、**完全禁止ではなく説明責任を課す**設計にする。

---

## 設計方針

| 方針 | 内容 |
|------|------|
| 常時強制 | Stop Hook で毎ターン発動。呼び忘れなし |
| パターンマッチなし | Claude が自己申告する方式。誤検知ゼロ |
| ブロックではなく要求 | 先送りを禁止するのではなく、理由・チケット番号の明示を要求 |
| チケットなしはユーザーに確認 | 自動作成ではなくユーザー判断に委ねる |

---

## アーキテクチャ

```
Claude が返答を終えて止まろうとする
        ↓
Stop Hook（~/.claude/hooks/nigecha-dameda.sh）が自動発動
        ↓
セルフチェックプロンプトを stdout に出力
        ↓
Claude が読んで自己申告

先送りなし       → そのまま完了
先送りあり
  チケットあり   → 番号・理由を明記して完了
  チケットなし   → ユーザーに「作成しますか？」と確認
```

## コンポーネント

### ① Hook スクリプト

**ファイル**: `~/.claude/hooks/nigecha-dameda.sh`

```bash
#!/bin/bash
cat << 'EOF'
【逃げちゃダメだチェック】
今の返答で「後続チケット」「後でやります」「次のPRで」「別途対応」と言った箇所があるか？

▶ 先送りがあった場合、以下を今すぐ確認すること：

  1. チケット番号（APP-XXXX）を明記したか？
     → 未明記なら今すぐ書くこと

  2. チケットがまだ存在しない場合：
     → 「新規チケットを作成しますか？」とユーザーに確認すること

  3. 先送りの理由を1行で書いたか？
     （スコープ外 / 設計変更が必要 / ユーザー合意あり のどれか）

▶ 先送りが一切なかった場合：
  → このチェックを無視してよい
EOF
```

### ② Hook 登録

**ファイル**: `~/.claude/settings.json`

```json
{
  "hooks": {
    "Stop": [
      {
        "hooks": [
          {
            "type": "command",
            "command": "bash ~/.claude/hooks/nigecha-dameda.sh"
          }
        ]
      }
    ]
  }
}
```

### ③ senju フック（配布用）

スキルではなくフックとして独立したディレクトリに配置する。  
SKILL.md ではなく README.md でインストール手順を提供する。

```
senju/hooks/nigecha-dameda/
├── README.md              ← インストール手順・判断基準
└── nigecha-dameda.sh      ← スクリプト本体
```

## 動作例

### ケース1: チケット番号なしで先送り（ブロック）

```
Claude: 「この修正は後続チケットで対応します。」
  → Hook 発動
  → Claude: 「チケット番号が未明記でした。APP-1999 で対応予定です。
             理由：現在のPRスコープ外のサービス変更が必要なため。」
```

### ケース2: チケットが存在しない（ユーザーに確認）

```
Claude: 「この修正は後続チケットで対応します。」
  → Hook 発動
  → Claude: 「チケットがまだ存在しません。
             新規チケットを作成しますか？」
  → ユーザーが判断
```

### ケース3: 正当な先送り（通過）

```
Claude: 「この修正は APP-1999 で対応します。
         理由：別サービスの設計変更が必要なため現PRスコープ外。」
  → Hook 発動 → Claude が自己チェック → 問題なし → 完了
```

### ケース4: 先送りなし（スルー）

```
Claude: 「修正しました。」
  → Hook 発動 → Claude が自己チェック → 先送りなし → 完了
```

## 先送り判断基準（参考）

| 状況 | 判定 |
|------|------|
| 変更量が50行以下 かつ 同じファイルを既に触っている | 今やれ |
| スコープ外のサービス・レイヤーに影響する | 先送り可（理由・チケット明記） |
| 設計判断が必要で決定権がユーザーにある | 先送り可（ユーザー確認必須） |
| チケットが存在しない | ユーザーに確認してから先送り |
| ユーザーが明示的に「後でいい」と言った | 先送り可 |

## 実装ステップ

1. `hooks/nigecha-dameda/nigecha-dameda.sh` を作成
2. `hooks/nigecha-dameda/README.md` を作成（インストール手順・判断基準）
3. `~/.claude/hooks/nigecha-dameda.sh` を配置（ローカル適用）
4. `~/.claude/settings.json` に Stop Hook を登録
5. 動作確認
6. senju にコミット
