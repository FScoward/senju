# codex-advisor — Codex を advisor として使う

Claude Code 純正の `advisor` ツールは、現在の会話履歴（タスク・実行したツール・観察結果・推論）を全部自動で
強いレビュアーモデルに転送して「独立した第二の目」を得る仕組み。

本スキルはその Codex 版。Codex CLI は会話履歴を勝手には見られないので、
**Claude 側が文脈を構造化して明示的に渡す** 必要がある。そのプロトコルを定義する。

> 「ちぃ、Claudeだけで考えてると同じ癖から抜けられない。Codexに渡すと、別の訓練分布から見た指摘が返ってくる」
> 「ユズキ：それが advisor の本質です。モデル冗長性、ではなくて、思考冗長性です」

---

## いつ発動するか（advisor 哲学の踏襲）

advisor は「呼ぶべきタイミング」が肝。以下の場面で必ず使う：

### 1. substantive work に入る「前」
- ファイル書き込み・編集・コミット・PR 作成の直前
- 解釈や設計判断を確定させる直前
- 大きな実装に着手する直前
- 探索・状況把握は substantive work ではないので、それを済ませた後に呼ぶ

### 2. 行き詰まった時
- 同じエラーが繰り返し出ている
- アプローチが収束していない
- 結果が想定と噛み合わない

### 3. アプローチを変えようとした時
- 「方針を変えるか」と迷った瞬間

### 4. 完了を宣言する「前」
- **必ず先に成果物を durable にしてから呼ぶ**（ファイル書き込み・コミット）
- advisor 呼び出し中にセッションが切れても成果物は残るように
- 「終わったと思う」だけで呼ばない

長めのタスクでは、最低でも「アプローチ確定前に1回」と「完了宣言前に1回」は呼ぶ。
短い反応的なタスク（直前のツール出力で次の手が自明な時）では毎回呼ばなくてよい。

## codex-cli との関係

| スキル | 役割 |
|--------|------|
| `codex-cli` | 任意プロンプトを Codex に投げる「汎用入口」 |
| `codex-advisor`（本スキル） | advisor プロトコル（構造化文脈渡し）を強制する「相談入口」 |

実行手段としては codex-cli と同じく `codex exec` を Bash で叩く。
違いは **渡すプロンプトの型が決まっていること**。

## 相談プロトコル（必須フォーマット）

Codex に渡すプロンプトは必ず以下の構造で組み立てる。
これを守ることで、Codex は「何を見て、何に答えればいいか」を即座に把握できる。

```
# 相談タイトル
（1行で、何について意見が欲しいか）

## ゴール
- ユーザーが最終的に達成したいこと
- 直近で達成しようとしている具体タスク

## 制約・前提
- リポジトリ / 言語 / フレームワーク
- 既存のコーディング規約・アーキテクチャ方針
- 動かしてはいけない部分、譲れない要件

## これまでに試したこと・観察したこと
- 実行したコマンド・読んだファイル・走らせたテストの要点
- そこから得られた事実（推測と事実は分けて書く）
- 失敗したアプローチがあれば「なぜ失敗したか」も

## 現状の方針 / 仮の結論
- 今 Claude が採用しようとしている案
- その案を選んだ理由

## 答えてほしいこと（ピンポイントで）
- 例：「この方針で進めて問題ないか。落とし穴があれば指摘してほしい」
- 例：「A と B で迷っている。判断基準と推奨を教えてほしい」
- 例：「完了宣言する前の最終チェック。見落としがあれば挙げてほしい」

## 関連ファイル（必要なら）
- パスと該当箇所の要点を箇条書きで
```

長すぎる場合は「現状の方針」と「答えてほしいこと」を最優先で残す。
ファイル本文を貼り付けるより、Codex が `read-only` サンドボックスで自分で読める前提でパスを渡す方が効率的。

---

## 実行コマンド

stdin を塞ぎ、本文だけを取り出して Claude 側で読む：

```bash
codex exec --sandbox read-only --color never \
  --output-last-message /tmp/codex-advisor.md \
  "$(cat <<'EOF'
# 相談タイトル
...

## ゴール
...

## 制約・前提
...

## これまでに試したこと・観察したこと
...

## 現状の方針 / 仮の結論
...

## 答えてほしいこと
...
EOF
)" < /dev/null
```

その後：

```
Read("/tmp/codex-advisor.md")
```

ファイル本文を Claude が読み、ユーザーへ要約 + 自分の適応方針を提示する。

## 受け取ったアドバイスの扱い方

advisor の出力は「真剣に受け止める」が原則。ただし盲従ではない。

1. **基本は採用する**: 自分が見落としていた観点・落とし穴・代替案は素直に取り込む
2. **実証的に矛盾したら適応する**: アドバイスに従って試して失敗した、あるいは一次ソース（ファイル・ドキュメント）が反証する場合は採用しない
3. **自分の調査結果と食い違う時は再相談**: 既に取得済みの証拠と Codex の助言がぶつかったら、黙って切り替えない。「自分は X を観測した、Codex は Y を提案、どちらの制約がタイブレイカーか」をもう一度 Codex に投げ直す（reconcile call）

ユーザーへの提示時は最低限：
- Codex が指摘した主要ポイント（箇条書き）
- そのうち採用するもの・しないもの・理由
- 次のアクション

---

## 最小フロー

1. substantive work の直前に立ち止まる（または stuck / 完了宣言前）
2. 完了宣言前なら、まず成果物を durable にする（書き込み・コミット）
3. 上記プロトコルでプロンプトを組み立てる
4. `codex exec --output-last-message /tmp/codex-advisor.md ... < /dev/null` で実行
5. `Read` で本文を読み、要点を抽出
6. ユーザーへ「Codex の指摘」「採用判断」「次の一手」を提示
7. 必要なら reconcile call で再相談

---

## アンチパターン

- 探索の最中に呼ぶ（まだ substantive work ではない）
- 成果物を保存せずに「完了前チェック」と称して呼ぶ（セッション切れリスク）
- プロトコルを守らずに「これどう思う？」だけ投げる（Codex が文脈不足で一般論を返す）
- ファイル全文をベタ貼りする（パスを渡して Codex に読ませる方が良い）
- アドバイスを盲従して観測済み事実と矛盾する方向に切り替える
- 1ターン内で何度も呼ぶ（advisor は思考の節目で呼ぶもの。連打しない）
