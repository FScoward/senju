# undine — 水先案内人

senjuのスキル群から**最適なスキルを提案**する案内役。  
ARIAの水先案内人（ウンディーネ）のように、ユーザーを正しい道へ静かに導く。

---

## 発火条件

### 明示トリガー（必ず発火）

- `undine` / `ウンディーネ`
- 「どのスキルを使えばいい？」「スキルを案内して」「おすすめは？」
- 「何から始めればいい？」「何を使えばいいか分からない」「スキルを教えて」

### 自動発火（以下を**すべて**満たす場合のみ）

1. ユーザーが開発タスクを依頼している
2. 特定のスキルを明示していない
3. 「〜したい」「〜するには？」「どうすれば」「何から」などの曖昧な表現がある
4. **別のスキルが既に実行中でない**（自己参照ループ防止）

---

## フロー

### Step 1: 意図の把握

ユーザーの発言から「何をしたいか」を抽出する。

**意図が不明確な場合** → 選択肢形式で2〜3択の質問をする（CLAUDE.mdの指示に従う）:

> やりたいことはどれ？
> A) Epic・US を設計〜PRまで一気に実装したい
> B) 既存コードのPRをレビュー・改善したい
> C) PRや機能を小さく分割・段階リリースしたい
> D) ドキュメント・設計書・要件定義書を作りたい
> E) コードの可視化・構造把握がしたい

**意図が明確な場合** → Step 2へ。

---

### Step 2: スキルカタログを動的収集

`skills/` 配下のすべての `SKILL.md` から frontmatter の `description` を読み込む:

```bash
find skills/ -name "SKILL.md" | sort | while IFS= read -r f; do
  dir=$(basename "$(dirname "$f")")
  desc=$(awk 'BEGIN{c=0} /^---/{c++;next} c==1{print} c==2{exit}' "$f" \
        | grep -A 10 'description:')
  echo "=== $dir ===" && echo "$desc" && echo
done
```

収集した各スキルの名前・説明・トリガー条件をすべて把握する。
**スキル一覧はファイルから動的に取得すること。**ハードコードしない。

---

### Step 3: マッチングと優先順位付け

ユーザーの意図と各スキルの description を照合してスコアリングする。

| マッチング観点 | 内容 |
|--------------|------|
| キーワード一致 | ユーザーの発言にスキルのトリガーワードが含まれるか |
| 目的一致 | スキルの目的がユーザーのゴールに合致するか |
| フェーズ一致 | 開発フローの現在地点（要件定義/設計/実装/レビュー）に合うか |
| 組み合わせ優位 | 単独より複数スキルの連携が有効なケースか |

---

### Step 4: 提案を出力

以下のフォーマットで出力する:

```
## おすすめスキル

**あなたの目的**: [ユーザーの意図を1行で要約]

---

### 第一候補: `スキル名`
**なぜ？**: [根拠を1〜2文で]
**起動方法**: `/スキル名`
**向いている状況**: [具体的な場面]

### 第二候補: `スキル名`（あれば）
...

### 第三候補: `スキル名`（あれば）
...

---

### 組み合わせパターン（有効な場合のみ）
例: `tidy-first` → `feature-flag-strategy` → `kouunryuusui`
[なぜその順序か、1〜2文で説明]

---

### 該当なし（スキルが合わない場合）
senjuにこれに合うスキルはまだない。
直接実装するか、新しいスキルの追加を検討して。
```

### Step 5: スキルを起動する

提案後、ユーザーが候補を選んだ（または「そのまま進めて」「やって」と言った）ら、
選ばれたスキルを **sub-agent または team** で起動する。

#### 単一スキルの場合 → sub-agent

選ばれたスキルの `SKILL.md` を Read で読み込み、エージェントに渡して実行する:

```
Agent({
  subagent_type: "oh-my-claudecode:executor",   // 複雑なフローは deep-executor
  prompt: "[読み込んだSKILL.mdの全内容]\n\n---\n\nユーザーの依頼: [元の依頼をそのまま記載]"
})
```

#### 組み合わせパターンの場合 → team pipeline

複数スキルを順番に実行する場合は team を使い、各スキルをステージとして渡す:

```
Agent({
  subagent_type: "oh-my-claudecode:executor",
  prompt: "以下の順でスキルを実行してください。\n
           Stage 1: [スキルA の SKILL.md 全内容]\n
           Stage 2: [スキルB の SKILL.md 全内容]\n\n
           ユーザーの依頼: [元の依頼]"
})
```

#### エージェント選択基準

| スキルの性質 | 使うエージェント |
|------------|---------------|
| 軽量・単純（`pr-size-guard`, `code-structure`, `jira-cli` 等） | `executor` |
| 複雑な統合フロー（`kouunryuusui`, `sier-dev`, `engineering-team-meeting` 等） | `deep-executor` |
| 複数スキルの組み合わせ | `executor` に順番を記述 or team |

> **注意**: ユーザーが「提案だけでいい」「自分でやる」と言ったら Step 5 はスキップする。

## スキル早見表（案内の補助用）

| やりたいこと | スキル |
|-------------|-------|
| Epic全体を自律実装（設計〜PR） | `kouunryuusui` |
| テストの充足性を3観点でレビュー | `mihari` |
| PRレビューを指摘ゼロまでループ | `review-loop` |
| Push前にPRサイズを確認 | `pr-size-guard` |
| 大きすぎるPRを事後分割 | `split-on-the-fly` |
| 依存チェーンのPRを管理 | `stacked-pr` |
| 機能追加を段階リリース（5PR） | `feature-flag-strategy` |
| 機能を縦スライスで段階設計 | `vertical-slice` |
| 整理と機能追加を分離 | `tidy-first` |
| DBスキーマを設計・変更 | `data-model-designer` |
| コードを図で可視化 | `code-structure` |
| ドキュメントを会話形式でレビュー | `doc-review-meeting` |
| スプリントプランニング/レトロ | `engineering-team-meeting` |
| 対話形式で要件定義書を作成 | `software-requirements` |
| ウォーターフォール設計書を生成 | `sier-dev` |
| JiraをCLIで操作 | `jira-cli` |

> この早見表は補助情報。実際の提案は必ずStep 2で動的収集したカタログを根拠にすること。
