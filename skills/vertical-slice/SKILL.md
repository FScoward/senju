# vertical-slice

機能を **BE/FE/DB で横に割る** のではなく、**薄いエンドツーエンドの縦スライス** として段階的に作るスキル。

## このスキルが解く問題

- BE/FE/DBを別PRに分けても、1スライスが大きすぎて結局レビュー困難
- BEだけ先に完成、FEは後 → **動作確認が遅延**してバグ発見が遅れる
- 「全AC一気に実装」で差分爆発、途中で設計ミスに気づいても引き返しが重い
- 仕様の全容を決めないと実装に入れない → スタートが遅れる
- 段階リリースしたいのに、設計が「全機能前提」で作られていて切り出せない

---

## 核心原則

1. **Walking Skeleton を最初に通す**: DB→BE→FE が1本だけ繋がる最小機能を**最初のPR**で作る
2. **各スライスは単独で価値がある**: 「データ保存できるけど表示できない」は NG。「1項目だけ保存できて表示もできる」は OK
3. **横ではなく縦に切る**: スライスの粒度は「機能の深さ」ではなく「機能の幅」
4. **次のスライスは前のスライスに積む、置き換えない**: 後方互換を保ちながら拡張する
5. **MVP で止まる勇気**: 全て作らなくてもリリース可能なら、そこで止めて次の機能に移る

## 水平分割 vs 垂直分割

### ❌ 水平分割（Horizontal Split）の典型失敗

```
Epic: ユーザー検索機能

PR-1: DB（検索用インデックス追加）         ← 動作確認不能
PR-2: BE（API実装、全フィルタ対応）         ← FE無しで動作確認不能
PR-3: FE（画面全部、全フィルタUI）          ← ← ← 結局ここが巨大化
```

**問題**:
- PR-2 と PR-3 がどうしても大きくなる（全フィルタを一度に実装するため）
- Walking Skeleton が遅い（PR-3 まで動作確認できない）
- 途中で仕様変更があると BE/FE の両方を改修する必要がある

### ✅ 垂直分割（Vertical Split）

```
Epic: ユーザー検索機能

Slice-1: Walking Skeleton — 名前1項目だけで検索（DB+BE+FE）
  ├─ DB: 既存テーブル、インデックスなし
  ├─ BE: 1APIのみ、名前LIKE検索
  └─ FE: テキストボックス1つ + 結果テーブル

Slice-2: 部署フィルタ追加（DB+BE+FE）
  ├─ DB: 部署への JOIN
  ├─ BE: 部署パラメータ追加
  └─ FE: 部署セレクトボックス追加

Slice-3: 役職フィルタ追加（DB+BE+FE）
  ├─ DB: 役職テーブル JOIN
  ├─ BE: 役職パラメータ追加
  └─ FE: 役職マルチセレクト追加

Slice-4: ソート・ページング（DB+BE+FE）
  ├─ DB: インデックス追加（パフォーマンス要件が出たら）
  ├─ BE: ソート・limit/offset パラメータ
  └─ FE: ソートヘッダー + ページネーションUI
```

**利点**:
- 各スライスが**単独で動く**。Slice-1 で既にユーザーが使える
- 各PRが小さい（1スライス = 1機能の深さ = 〜200行）
- 途中で仕様変更があっても、そこまでのスライスは捨てずに済む
- MVP で止める選択肢が常にある

## Walking Skeleton の設計

最初のスライスは **最も痩せたエンドツーエンド**にする:

| 要素 | Walking Skeleton で含む | 次のスライスで追加 |
|------|---------------------|-----------------|
| DB | 既存テーブルのみ、新規スキーマ変更なし（可能なら） | 新テーブル、インデックス |
| API | 1エンドポイント、1パラメータ、1レスポンス形式 | パラメータ追加、エラーハンドリング |
| バリデーション | 最低限（必須チェックのみ） | 詳細バリデーション |
| UI | 最小操作（1ボタン、1結果表示） | 装飾、UX改善、状態保持 |
| エラー処理 | エラー時は「エラー」とだけ表示 | 詳細メッセージ、リトライ |
| 権限 | 1ロールのみ対応 | 複数ロール、細かい権限 |
| ログ | 最小限 | 詳細ログ、メトリクス |
| テスト | 正常系1つ | 境界値、異常系、E2E |

**鉄則**: Walking Skeleton で **実ユーザーの1シナリオが完走できる**こと。社内のテストユーザーでもよい。

## スライス分割のパターン

### パターン1: 機能軸（Feature Flags）

機能を「必須」「重要」「任意」に分けて、必須から順にスライス化:

```
Epic: 申請機能
├─ Slice-1: 申請作成 + 一覧表示 (MoSCoW: Must)
├─ Slice-2: 申請の承認フロー (Must)
├─ Slice-3: 通知機能 (Should)
├─ Slice-4: 添付ファイル (Could)
└─ Slice-5: 検索・フィルタ (Could)
```

### パターン2: データ軸

扱うデータ種類を1つずつ増やす:

```
Epic: レポート機能
├─ Slice-1: 1種類のレポート（売上）を1画面で表示
├─ Slice-2: 2種類目（顧客数）を追加
├─ Slice-3: フィルタ期間指定を追加
└─ Slice-4: エクスポート機能を追加
```

### パターン3: ユーザー軸

対応ユーザーロールを段階的に広げる:

```
Epic: ダッシュボード
├─ Slice-1: 管理者ロール向けのみ実装
├─ Slice-2: 一般ユーザー向けの制限版を追加
└─ Slice-3: 外部ユーザー向け（認証別）
```

### パターン4: シナリオ軸（Golden Path First）

最頻パス（Golden Path）を先に作り、エッジケースを後から:

```
Epic: 決済機能
├─ Slice-1: クレジットカード決済の成功ケース（正常系のみ）
├─ Slice-2: カード拒否・タイムアウトのエラー処理
├─ Slice-3: 別決済手段（銀行振込）追加
└─ Slice-4: 返金・キャンセル処理
```

## ワークフロー

### Step 1: Epic を Walking Skeleton + スライスに分解

1. Epic の AC を全て書き出す
2. 最小の「エンドツーエンドで動く1シナリオ」を特定 → **Walking Skeleton**
3. 残りの AC を機能/データ/ユーザー/シナリオのいずれかの軸で3〜6スライスに分割
4. 各スライスに**明確な完了条件**（ACサブセット）を付ける

### Step 2: スライス優先度の決定

MoSCoW法で優先度を付ける:

| 優先度 | スライスの扱い |
|-------|---------------|
| Must | 必ず実装する。Walking Skeleton は必ず Must |
| Should | 通常実装するが、時間なければ次スプリント |
| Could | あれば嬉しい。リリース後にユーザーフィードバックで判断 |
| Won't | 今回はやらない。将来チケット化 |

**MVP定義**: Must スライスが全て終わった時点 = MVP。ここで**リリース可能**と判断する。

### Step 3: チケット化

各スライスを独立したチケットにする:

```markdown
Epic: {機能名}

## スライス一覧

### Slice-1: {タイトル} [Walking Skeleton]
- **AC**: {エンドツーエンドで動く1シナリオ}
- **含まれる**: DB変更（最小）、BE 1API、FE 1画面
- **含まれない**: フィルタ、エラー詳細化、エッジケース
- **期待PRサイズ**: 〜300行

### Slice-2: {機能追加}
- **AC**: {Slice-1 に追加する価値}
- **依存**: Slice-1
- **期待PRサイズ**: 〜200行

...
```

### Step 4: スライス実装

各スライスは独立してマージ可能なPRとして実装:

- **Slice-N** は Slice-(N-1) のマージ後に base=main で出す
- 並列可能なら Stacked PR（`stacked-pr` スキル参照）
- 各スライスの実装前に `pr-size-guard` の閾値を意識する

### Step 5: 各スライスのマージ後にMVP判定

Must スライスが全て完了した時点で:

- [ ] リリース可能か？（受け入れテスト、ステークホルダー確認）
- [ ] Should/Could に進むべきか、次のEpicに移るべきか？

**MVP で止める選択肢を常に持つ**ことで、過剰開発を防ぐ。

## feature-flag-strategy との組み合わせ

vertical-slice は「機能の段階設計」、feature-flag-strategy は「段階リリース」。両者は直交する。

### 組み合わせ例

```
Epic: ユーザー検索機能（フラグで隠しながら段階実装）

1. [FF-ADD] 検索機能フラグ定義追加
2. [VS-1] Walking Skeleton（名前検索のみ、flag behind）
3. [VS-2] 部署フィルタ追加（flag behind）
4. [VS-3] 役職フィルタ追加（flag behind）
5. [FF-ENABLE-DEV] dev環境でフラグON
6. [VS-4] ソート・ページング（flag behind、UIブラッシュアップ）
7. [FF-ENABLE-STAGING] staging環境でフラグON
8. [FF-ENABLE-PROD-CANARY] 本番5%でフラグON
9. [FF-ENABLE-PROD-100] 本番100%
10. [FF-REMOVE] フラグ削除
```

各 VS-N は Walking Skeleton の上に積まれ、フラグで本番ユーザーからは隠される。**小さく作って、小さく検証して、全部できたら有効化**。

## アンチパターン

### ❌ NG: Walking Skeleton を飛ばして大きなSliceから始める

「Slice-1 = 全部実装」は vertical-slice ではない。

**対策**: Slice-1 は必ず「動く最小機能」。迷ったらさらに削る。

### ❌ NG: スライスが「水平」になっている

Slice-1: DB 全部, Slice-2: BE 全部, Slice-3: FE 全部 — これは水平分割。

**対策**: 各スライスに DB+BE+FE が全部含まれる（変更が無い層があってもよい、が「1層だけ」は NG）。

### ❌ NG: スライスが互いに依存しすぎる

Slice-2 が Slice-1 の内部実装に依存し、Slice-1 変更で Slice-2 も壊れる。

**対策**: 各スライスは**ユーザー価値**として独立。内部APIは Slice-1 で安定化する。

### ❌ NG: 後方互換を壊す変更

Slice-2 で Slice-1 の API を破壊的変更する → Slice-1 の既存動作が壊れる。

**対策**: Slice-N は Slice-(N-1) に**追加**する。置き換えるなら Expand & Contract（data-model-designer）を併用。

### ❌ NG: スライスが多すぎる（10本以上）

細かすぎてオーバーヘッドが爆発する。

**対策**: 3〜6スライスが実用域。多すぎるなら Epic 自体が大きすぎる → Epic を分ける。

### ❌ NG: Walking Skeleton が1スプリントに収まらない

「最小」と言いつつ実装2週間。それは Walking Skeleton ではない。

**対策**: Walking Skeleton は数日（多くて1週間）で完了する粒度まで削る。

## 他スキルとの関係

| スキル | 役割 |
|--------|------|
| **vertical-slice** | 機能そのものを段階設計（MVP → 拡張） |
| `feature-flag-strategy` | 段階リリース（フラグで隠しながら） |
| `tidy-first` | 各スライスの前にリファクタを先行PR化 |
| `stacked-pr` | スライスが依存チェーン状になった時の運用 |
| `pr-size-guard` | 各スライスのPRサイズを事後検証 |
| `data-model-designer` | スライス内のDB変更を Expand & Contract で安全に |
| `kouunryuusui` | Epic→スライス分割の E3 で本スキルを使用 |

## チェックリスト

### Epic 分解時
- [ ] Walking Skeleton を特定した（エンドツーエンドで動く最小機能）
- [ ] 残りを機能/データ/ユーザー/シナリオのいずれかの軸でスライス化した
- [ ] スライスは3〜6本に収まっている
- [ ] 各スライスに MoSCoW 優先度を付けた
- [ ] MVP を定義した（Must スライス完了時点）

### スライス設計時
- [ ] 各スライスが**単独でユーザー価値を持つ**
- [ ] 各スライスに DB+BE+FE が含まれる（どれかが変更なしでもOK、だが「1層だけ」はNG）
- [ ] 各スライスが後方互換を保っている
- [ ] Walking Skeleton が数日〜1週間で完了する粒度

### 実装時
- [ ] Slice-1 をマージしてから Slice-2 に進む
- [ ] 各スライスのPRサイズが 〜300行に収まる
- [ ] MVP時点で「リリース可能か」を判定する
