# チケットドラフトテンプレート

SKILL.md の Step 4b で使用するテンプレート。
`docs/06_wbs/ticket-drafts/TICKET-NNN-xxx.md` として保存する。

## 役割分担の原則

このドラフトは **kouunryuusui T0 が `refine-ticket` に渡す入力**として設計されている。

| 記入者 | 記入タイミング | 記入内容 |
|--------|-------------|---------|
| **spec-to-plan**（このスキル） | WBS生成時（設計書が揃った状態） | ユーザー価値・優先度・依存・draft GWT |
| **refine-ticket**（kouunryuusui T0） | 実装着手前（コードベース参照可能） | Examples・異常系・影響範囲・Then句の具体化 |

**見積**: {X日}
**依存チケット**: {TICKET-NNN / なし}

---

## ユーザー価値

- **体験の変化**: {この機能がないときとあるときで何が変わるか（1〜2行）}

## draft AC

> **Given/When のみ記入。Then の具体値・Examples は T0（refine-ticket）が補完する。**

### AC-1: {正常系の概要}

**Given** {事前条件（ユーザーのログイン状態・データの存在等）}
**When**  {ユーザーの操作（ボタンクリック・API呼び出し等）}
**Then**  {結果の概要}（※具体値・機械判定可能な形式は T0 で補完）

### AC-2: {別の正常系 or 主要な分岐（あれば）}

**Given** ...
**When**  ...
**Then**  ...（T0 補完）

## T0（refine-ticket）で補完が必要な項目

以下は **意図的に空けている**。コードベース調査後に refine-ticket が補完する。

- [ ] **Then 句の機械判定可能な具体値**（「正しく表示される」→ HTTP 200 + フィールド名・値の明示）
- [ ] **Examples**（境界値・具体的な入力データと期待値）
- [ ] **異常系 AC**（バリデーションエラー・権限エラー・DB制約違反等）
- [ ] **影響範囲調査**（呼び出し元/先・既存テスト・横断的関心事の grep）
- [ ] **業務フロー整合性**（状態遷移・前後工程との整合）— Tier 2/3 のみ
- [ ] **非機能要件 AC**（パフォーマンス・セキュリティ）— 設計方針書で言及された場合

## 記入例

```markdown
# TICKET-003: ユーザー登録フォームの実装

> **draft** — T0（refine-ticket）での補完が必要な項目は末尾を参照

**Tier**: 2
**優先度**: P1 High（スコア: 8/12）
**見積**: 3日
**依存チケット**: TICKET-001（認証基盤）

---

## ユーザー価値

- **対象ユーザー**: 新規登録ユーザー（未認証の一般ユーザー）
- **体験の変化**: アカウントなしでは使えなかったサービスを、メールとパスワードで即座に利用開始できる
- **Why Now**: v1.0 リリースのブロッカー。登録機能なしでは他の全機能が利用不可

## 開発者価値

- **技術的改善**: 認証基盤（TICKET-001）の検証を兼ねる。UserRepository の CRUD が確立されることで後続の管理機能チケットがスムーズに進む

---

## 優先度スコア根拠

| 軸 | スコア | 根拠 |
|----|--------|------|
| ユーザー影響度（0〜3） | 3 | 全ユーザーがこの機能を必ず通る |
| ビジネス緊急度（0〜3） | 3 | v1.0 リリースのブロッカー |
| 技術リスク（0〜3）     | 1 | 既知のパターン。UserEntity が既に設計済み |
| 依存関係（0〜3）       | 1 | TICKET-004（プロフィール編集）が待ち状態 |
| **合計**             | **8/12** | |

---

## draft AC

### AC-1: 正常系 — 有効な情報で登録できる

**Given** 未ログインのユーザーが登録ページを開いている
**When**  有効なメールアドレス・パスワードを入力して「登録」ボタンを押す
**Then**  アカウントが作成されてログイン済み状態になる（T0 補完）

### AC-2: メール重複時のエラー表示

**Given** 既に同じメールアドレスで登録済みのアカウントが存在する
**When**  同じメールアドレスで登録を試みる
**Then**  エラーメッセージが表示されてアカウントは作成されない（T0 補完）

---

## T0（refine-ticket）で補完が必要な項目

- [ ] Then 句の具体値（HTTP ステータス・レスポンスフィールド・リダイレクト先等）
- [ ] Examples（パスワードの最短・最長文字数、メールフォーマットの境界値等）
- [ ] 異常系 AC（パスワード強度不足・メール形式不正・DB タイムアウト等）
- [ ] 影響範囲調査（UserRepository・AuthService・メール送信処理の grep）
- [ ] 業務フロー整合性（登録後メール認証フローとの接続確認）

---

## 設計書参照

- 要件定義書: 「3.2 ユーザー登録機能」
- 基本設計書: 「4.1 認証フロー」「5.3 User エンティティ」
- 外部設計書: 「画面 S-003 登録フォーム」「API POST /users」
```

## 命名規則まとめ

```
TICKET-{3桁連番}-{機能を表す動詞句（ケバブケース）}.md

良い例:
  TICKET-001-setup-auth-foundation.md      ← 動詞で始まる
  TICKET-002-implement-product-list-api.md ← 機能が明確
  TICKET-003-add-user-registration-form.md ← 1チケット1機能

悪い例:
  TICKET-001-auth.md                       ← 動詞なし・曖昧
  TICKET-002-backend.md                    ← 粒度が大きすぎる
  TICKET-003-fix.md                        ← 何のfixか不明
```

## Tier別の draft AC 詳細度

| Tier | Given/When の詳細度 | AC 件数の目安 |
|------|--------------------|-------------|
| Tier 1 | 1文で十分 | AC 1〜2件 |
| Tier 2 | 状態・前提条件まで記載 | AC 2〜4件（正常系 + 主要な分岐） |
| Tier 3 | ロール・データ状態・前後工程も記載 | AC 3〜6件（T0 補完前提で多めに洗い出す） |
