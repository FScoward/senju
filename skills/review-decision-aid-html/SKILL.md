# レビュー対応 Decision Aid HTML スキル

PRレビューの指摘を受け取り、各指摘に対して「今回対応 / スキップ / 後続化 / レビュアー確認」のいずれかを選択できるHTMLページを生成する。

## このスキルが解くこと

- 各指摘に「対象コード・指摘の意味・必要な理由・放置リスク・推奨判断」を揃えて提示する
- 各項目の直下に選択肢ラジオボタンを置く（最後にまとめて選ばせない）
- 最後に全選択結果を1クリックでコピーできるテキストにまとめる

---

## Step 1: 入力を解析する

入力は以下のいずれか：

1. receive-review スキルの出力テキスト（SHOULD/NICE/DISCUSS/DEFER 分類済み）
2. 生のレビューコメント一覧（未分類なら自分で SHOULD/NICE/DISCUSS/DEFER に分類する）
3. ユーザーが手入力した指摘リスト

各指摘について以下を確定させてから HTML 生成に進む：

| 項目 | 内容 |
|------|------|
| 分類 | SHOULD / NICE / DISCUSS / DEFER |
| ID | S1, N1, D1, F1 形式（S=SHOULD, N=NICE, D=DISCUSS, F=DEFER） |
| タイトル | 「何のファイルの何についての話か」が分かる日本語。英語ラベル1単語は禁止 |
| 対象 | ファイルパス または クラス名/メソッド名 |
| 指摘の意味 | レビュアーが何を言っているか（1〜3文） |
| なぜ必要か | 放置した場合にどんな問題が起きるか |
| 推奨判断 | このスキルとして推奨する選択肢（理由付き） |

**DISCUSS の場合は追加で以下を確定させる：**

| 項目 | 内容 |
|------|------|
| A案 | 現状の実装・または選択肢1。1〜2文で説明 |
| A案のトレードオフ | メリット・デメリット各1文 |
| B案 | 提案内容・または選択肢2。1〜2文で説明 |
| B案のトレードオフ | メリット・デメリット各1文 |
| Claude の推奨案 | A案/B案のどちらを推すか、理由とともに明示する。「レビュアー次第」は禁止。Claude自身の判断を出す |
| 確認コメント案 | そのままGitHubに投稿できる文章。A案/B案を示してレビュアーの意図を確認する内容 |

---

## Step 2: HTMLを生成する

`tmp/pr-{PR番号}-review-decision-aid.html` に保存。PR番号が不明な場合は `tmp/review-decision-aid.html`。

**HTMLは単体ファイルで完結**させること。外部ライブラリ・CDN・フォントファイルの参照は禁止。CSS/JS はすべて `<style>` / `<script>` タグに埋め込む。

### ページ構成（この順序で）

1. **Hero セクション** — タイトル・統計カウント（未解決/今回対応候補/要確認/後続化候補）・推奨進め方
2. **Context セクション** — `<details>` 折りたたみ。目的・入力ソース・スコープ
3. **SHOULD セクション** — 各指摘をカード形式で。id/title/file/詳細(折りたたみ)/ラジオ選択肢。デフォルト「今回対応」。選択肢: `今回対応 / 後続化 / スキップ / その他（自由入力）`
4. **NICE セクション** — 同構造。デフォルト「スキップ」。選択肢: `スキップ / 今回対応 / 後続化 / その他（自由入力）`
5. **DISCUSS セクション** — 各カードに以下を含める（すべて常時表示、折りたたみ禁止）:
   - A案/B案の比較ブロック（それぞれの説明とトレードオフ）
   - Claude の推奨案と理由
   - 確認コメント案（編集可能テキストエリア＋コピーボタン）
   - ラジオ選択肢。デフォルト「レビュアー確認」。選択肢: `レビュアー確認 / A案で対応 / B案で対応 / 対応不要返信 / その他（自由入力）`
   - 状態遷移が絡む複雑な論点の場合のみ SVG 図を入れる（全 DISCUSS に必須ではない）
6. **DEFER セクション** — 同構造 + ロードマップSVG。デフォルト「後続化」。選択肢: `後続化 / 今回対応 / 対応不要 / その他（自由入力）`
7. **Copy セクション** — 全選択を集約するtextareaとコピーボタン。ラジオ変更時と `window.load` 時に自動更新

HTMLテンプレート・CSS・JS・SVGの実装詳細は `references/implementation.md` を参照すること。

## Step 3: SVG図

SVG図は **A案/B案の比較で状態遷移が絡む場合のみ** 入れる。すべての DISCUSS に必須ではない。装飾ではなく、状態遷移・ロードマップの説明のみに使う。
SVGテンプレートは `references/implementation.md` を参照。

## 説明文の書き方ルール（最重要）

**読む人は「判断したい人」であって「実装を理解したい人」ではない。** 読んで3秒で意味がつかめる文章にする。

- **一文を短く**。3つ以上繋がったら分割する
- **英語技術用語は括弧で言い換え**を添える
- **「なぜ問題か」を具体的に** — 「設計上の懸念」は説明になっていない。「将来Xを変えると壊れる」まで書く
- **放置リスクは「何が起きるか」を書く** — 「問題が発生する可能性」は説明になっていない

NG/OK 例:

| NG（頭に入らない） | OK（3秒で分かる） |
|-------------------|-----------------|
| claim作成とCloud Tasks enqueueは同一transactionにできないため補償処理が必要です | DB保存とタスク登録は一緒に取り消せない。タスク登録だけ失敗すると「処理中」のまま宙に浮く |
| QueryModelがDomain Entityを直接参照しているのは設計上の懸念 | 検索処理がドメインロジックに直接触れている。将来ドメイン側を変えると検索も壊れやすくなる |

## 禁止事項

| 禁止 | 理由 |
|------|------|
| 英語ラベル1単語だけのタイトル | どのコードの話か分からない |
| 最後だけに選択肢 | 各項目を読みながら選べない |
| 「なぜ必要か」の省略 | NICE以上の指摘は理由があって出ている |
| 状態遷移を長文テキストだけで説明 | 視覚化しないと境界条件が伝わらない |
| コピー欄の省略 | 次の作業指示につなげられない |
| 外部CSS/JS依存 | オフライン・セキュアな環境で使えなくなる |
| 技術用語を言い換えなしで使う | 判断したい人が3秒で分からない文章になる |

## 完成チェックリスト

- [ ] 各レビュー項目に個別のラジオボタン選択肢がある（「その他（自由入力）」を含む）
- [ ] タイトルが「何のファイルの何についての話か」分かる日本語になっている
- [ ] 各項目に「対象ファイル」「指摘の意味」「なぜ必要か」「放置リスク」が書かれている
- [ ] DISCUSS カードに A案/B案の比較ブロックがある
- [ ] DISCUSS カードに Claude の推奨案と理由が常時表示されている
- [ ] DISCUSS カードに確認コメント案（コピー可）が常時表示されている
- [ ] DISCUSS 選択肢が `レビュアー確認 / A案で対応 / B案で対応 / 対応不要返信` になっている
- [ ] 状態遷移が絡む DISCUSS のみ SVG 図がある（全 DISCUSS に必須ではない）
- [ ] 詳細情報は `<details>` で折りたたまれている
- [ ] 最後にコピー欄があり、ラジオ変更で自動更新される
- [ ] ファイルを `tmp/` 配下に保存した
