# ソフトウェア要求エンジニアリング $ARGUMENTS

ソフトウェア要求 第3版の手法を使って、**対話形式で要求を引き出し → 整理 → Markdown仕様書を出力**する。

## 要求の3層構造

```
ビジネス要求  ← なぜこのシステムが必要か（目的・目標・KPI）
    ↓
ユーザー要求  ← ユーザーが何を達成したいか（ゴール・タスク）
    ↓
機能要求     ← システムが何をすべきか（具体的な振る舞い）
```

加えて：
- **非機能要求（品質特性）**: パフォーマンス、セキュリティ、使いやすさ等
- **制約**: 技術的・業務的・法的制約

---

## 進め方：6フェーズ

### Phase 1: コンテキスト把握

最初の質問（1〜2個に絞る）：

- 「どんなシステム／機能を作りたいですか？」
- 「誰が使いますか？」

回答を受けたら要約して「〜ということですね？」と確認する。

### Phase 2: ビジネス要求の明確化

- このシステムで解決したいビジネス上の問題は何か？
- 成功の基準は何か？（指標・KPI）
- スコープ内・スコープ外の境界はどこか？

**なぜ（Why）を問い続けることが最重要**。背景を理解することで、より良いソリューションを提案できる。

### Phase 3: ステークホルダーとユーザー要求

- 誰がこのシステムを使うか（ユーザーカテゴリの列挙）
- 各ユーザーが達成したいゴールは何か
- ユーザーストーリー形式で整理する：

```
〜として、
〜したい、
なぜなら〜
```

### Phase 4: 機能要求の洗い出し

ユーザー要求をブレークダウンして機能要求に変換する。

- 「〜できること」という形で記述
- MoSCoW法で優先度をつける：
  - **Must**: 必須（これがなければシステムとして成立しない）
  - **Should**: 重要（あるべきだが、なくても動く）
  - **Could**: あれば嬉しい（時間・予算があれば）
  - **Won't**: 今回はやらない（将来の候補）

### Phase 5: 非機能要求

必要に応じて確認する（全部聞かなくてよい）：

| 観点 | 例 |
|------|-----|
| パフォーマンス | 応答時間、同時接続数 |
| セキュリティ | 認証方式、データ保護 |
| 使いやすさ | UIガイドライン、アクセシビリティ |
| 信頼性 | 可用性、障害復旧時間 |
| 移行・互換性 | 既存システムとの連携 |

### Phase 6: ドキュメント出力

十分な情報が集まったら「仕様書を出力しますか？」と確認し、以下のテンプレートで出力する。

## 出力テンプレート

```markdown
# ソフトウェア要求仕様書
**プロジェクト名**: {名前}
**作成日**: {日付}
**バージョン**: 1.0

---

## 1. プロジェクト概要
{概要を2〜3文で}

## 2. ビジネス要求
### 2.1 背景・問題
{解決したい課題}

### 2.2 目標・成功基準
{KPI・達成目標}

### 2.3 スコープ
**スコープ内**: ...
**スコープ外**: ...

## 3. ステークホルダー
| 役割 | 説明 | 関心事 |
|------|------|--------|
| ... | ... | ... |

## 4. ユーザー要求（ユーザーストーリー）
### US-001: {タイトル}
- **ユーザー**: {ロール}として
- **ゴール**: {やりたいこと}したい
- **理由**: なぜなら{理由}
- **受入基準**:
  - [ ] {条件1}
  - [ ] {条件2}

## 5. 機能要求
### Must（必須）
- FR-001: {機能の説明}

### Should（重要）
- FR-xxx: ...

### Could（任意）
- FR-xxx: ...

### Won't（対象外）
- FR-xxx: ...

## 6. 非機能要求
### 6.1 パフォーマンス
- {要求}

### 6.2 セキュリティ
- {要求}

### 6.3 その他
- {要求}

## 7. 制約
- {技術的・業務的・法的制約}

## 8. 用語集
| 用語 | 定義 |
|------|------|
| ... | ... |
```

## 対話のルール

1. **一度に質問は1〜3個まで** — 多すぎると圧倒される
2. **回答を要約して確認する** — 「〜ということですね？」
3. **矛盾・曖昧さを見つけたら優しく指摘** — 「〜と〜が矛盾しているかもしれません」
4. **「他に〜はありますか？」で深堀り** — 見落としを防ぐ
5. **完璧を求めない** — 要求は変わる。今わかっていることを整理すれば十分

> ステークホルダーの言葉で書く。技術用語ではなくビジネス・ユーザーの言葉で要求を記述することで、全員が理解できる仕様書になる。
