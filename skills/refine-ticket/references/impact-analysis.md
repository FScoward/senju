# 影響範囲の洗い出し（コード波及）

AC を磨く前に、対象チケットの変更が **どこに波及するか** を素早く洗い出すための手順。
ここで得た情報は AC の補完（特に異常系・境界値・副作用・回帰）と、出力ファイルの
「影響範囲レポート」セクションに反映する。

> **位置づけ**: refine-ticket の責務は AC 文書の品質。網羅的な設計調査ではない。
> ここでは「AC を機械判定可能にするために必要な情報を集める軽量調査」を行う。
> 徹底的な設計レベルの影響範囲調査は `kouunryuusui` E2a で実施する。

---

## いつ実施するか

| 状況 | 実施する？ |
|------|---------|
| 既存修正（バグ修正・仕様変更・機能拡張・リファクタリング） | **必須** |
| 新規開発（既存コードに該当機能が存在しない） | 軽量（後述「新規開発時」）|
| Tier 1（CRUD・設定変更・変更ファイル3未満） | 軽量で OK |
| Tier 2/3 | 全7観点を実施 |

---

## 7観点チェックリスト

各観点について「**誰が・何を・なぜ参照しているか**」を short list で集める。網羅ではなく **AC 補完に必要な分** を見つけるのが目的。

### 1. 呼び出し元（Upstream callers）

修正対象の **クラス / 関数 / API エンドポイント / DB カラム** を、誰がどこから呼んでいるか。

**収集コマンド例:**
```bash
# シンボル参照
rg -n "ClassName|methodName" --type=kt --type=ts
ast-grep -p 'methodName($$$)' --lang kotlin
# API エンドポイント
rg -n '"/api/path"' --type=ts
# DB カラム
rg -n 'column_name' src/main/resources/db/ infrastructure/
```

**AC への反映**: 既存呼び出し元の振る舞いが変わらないことの回帰 AC を追加する。

### 2. 呼び出し先（Downstream dependencies）

修正対象が呼び出している外部依存（DB / 別 UseCase / 外部 API / バッチ / キャッシュ / 通知）。

**AC への反映**: 依存が落ちた時の異常系 AC（タイムアウト・エラーレスポンス・部分失敗）を追加する。

### 3. データフロー（Read/Write 範囲）

修正対象が **読み書きするテーブル・カラム** が他機能でどう使われているか。

- 同じテーブルを書く別 UseCase はないか（同時更新・整合性リスク）
- 同じカラムを読む集計バッチ・エクスポート・レポート・キャッシュはないか
- マスキング・暗号化対象カラムが含まれていないか

**AC への反映**: 副作用 AC（DB 状態変化）+ 並行更新・テナント分離の AC を追加する。

### 4. 型・契約・スキーマ（Contracts）

- API レスポンス型変更が **FE 側 zod スキーマ / 型定義 / hooks** に波及していないか
- enum 追加・変更が **switch / when の網羅性** を壊さないか
- DB スキーマ変更が **既存データ・FK・インデックス** に波及しないか

**AC への反映**: 型互換性・enum 網羅性・バリデーション AC を追加する。

### 5. 既存テスト（Test fallout）

変更箇所をカバーする既存テスト一覧を出す。

```bash
rg -n -l "ClassName|methodName" src/test/
```

- 期待値が変わるテストはどれか
- モック設定の更新が必要なテストはどれか
- 削除すべきテスト（旧仕様検証）はあるか

**AC への反映**: 「既存テストの期待値変更点」を補完履歴ログに記録。回帰 AC として追加。

### 6. 横断的関心事（Cross-cutting concerns）

- **アクセス制御**: テナント分離 / ロール別ガード / IDOR 防止が変わるか
- **監査ログ**: 操作ログ追加 / 変更が必要か
- **メトリクス**: ダッシュボード / アラート対象が変わるか
- **トランザクション境界**: 既存 transaction { } の中か外か、ネストするか
- **キャッシュ**: invalidate が必要か、stale risk があるか
- **通知**: メール / Slack / Push の追加・抑制が必要か

**AC への反映**: 該当する横断的関心事ごとに副作用 AC・非機能 AC を追加する。

### 7. リリース順序・互換性（Release & compatibility）

- BE / FE / DB マイグレーション / Terraform / batch のデプロイ順序
- 旧クライアント・既存 in-flight データとの互換性
- feature flag / E&C / stacked PR の必要性

**AC への反映**: リリース順序 / ロールバック手順を AC 外の前提として明記。
E&C 必須なら呼び出し元（kouunryuusui E3）へ申し送る。

## 新規開発時（軽量版）

該当機能が既存にない場合は以下に絞る:

1. **隣接機能との重複・統合**: 似た機能が既に存在しないか（grep / シンボル検索 1 回）
2. **共通基盤への参加**: ベースクラス・共通 UseCase・共通 hook を継承/利用する場合の契約
3. **DB スキーマ**: 新テーブル / 既存テーブル拡張のどちらか。後者は破壊的変更チェック

## 出力: 影響範囲レポート

`refine-ticket` の出力ファイルに以下のセクションを追加する。**全観点を等しく書く必要はない**。
該当なしは「該当なし」と明記し、判断したことを示す（沈黙は「見落とし」と区別が付かない）。

```markdown
## 影響範囲レポート

### 1. 呼び出し元
- `EPAlertManagementController:97` — `?: fallback` を `requireNotNull` に変える AC で影響
- `use-fetch-ep-alert-list.ts:42` — レスポンス型変更で zod 更新が必要

### 2. 呼び出し先
- `EPAlertRepository.listBy` — projectId バリデーション追加で例外契約が変わる

### 3. データフロー
- `ep_alert_flags` テーブル: 集計バッチ `EPAggregationBatch` が読んでいる → 並行更新を AC4 に追加

### 4. 型・契約・スキーマ
- `EPAlertState` enum: 新値 `CLOSED` 追加。FE `ALERT_STATE_META` / BE `when` 網羅性チェック

### 5. 既存テスト
- `EPAlertManagementControllerTest.kt`: 期待値更新（5 ケース）
- `use-fetch-ep-alert-list.test.ts`: モック更新（2 ケース）

### 6. 横断的関心事
- アクセス制御: テナント分離既存。projectId 検証追加で IDOR ガード強化（AC5）
- 監査ログ: 既存。変更なし
- トランザクション: 新規 transaction 不要（既存内で動く）

### 7. リリース順序
- BE → FE の順。FE は旧 enum でも動く後方互換を保つ（DR-001）
- E&C 該当なし
```

## アンチパターン

| アンチパターン | 問題 | 改善 |
|---------------|------|-----|
| 「影響範囲: なし」とだけ書く | 7観点を見たか不明 | 観点ごとに「該当なし」を明記 |
| 「全部見ました」と書くだけで具体 0 | 検索コマンドの結果が無い | grep / find references のヒットを書く |
| FE 側だけ見て BE 側を見ない | レイヤー横断の波及を見落とす | BE / FE / DB / 設定 / バッチ を全部スキャン |
| 「動けば大丈夫」 | 副作用・並行更新・キャッシュ stale を見落とす | 観点 6（横断的関心事）を明示的に潰す |

---

## 補完履歴ログとの関係

- 影響範囲調査で **AC に推定・補完を入れた** 場合は、SKILL.md の「補完履歴ログ」にも記録する
- 影響範囲レポート自体は事実の収集（grep 結果・参照箇所）であり、推定ではない
- ただし「この呼び出し元には変更影響なし」と判断した根拠は影響範囲レポートに残す
