# 周辺コンテキストの補完

チケット本体だけでは「何を作ればよいか」判断できない場合に、AC を磨く前に
周辺情報を集めるための手順。`refine-ticket` のフロー先頭（存在チェック）で使う。

---

## Step 1: 存在チェック

チケットの現状を確認し、次のアクションを決める。

| チケット状態 | アクション |
|---|---|
| AC が存在する | → フォーマット修正へ（`ac-completion.md` / `ac-guidelines.md`）|
| AC なし だが仕様書/PRD リンクあり | **仕様書→US+AC 自動生成**（`ac-completion.md`）→ フォーマット修正へ |
| AC なし かつ 仕様書なし かつ 説明が十分 | **`sier-dev` スキルで要件定義書を生成**（任意・インストール時）→ US+AC 自動生成 → フォーマット修正へ |
| AC なし かつ 仕様書なし かつ 説明が不十分（3行未満） | **→ Step 1a（周辺チケット参照）** → コンテキスト補完後に再判定 |

> **⚠️ チケット内容が不十分な場合**は、Step 1 判定の前に Step 1a を実施する。
> 「内容が不十分」とは：説明が3行未満 かつ AC なし かつ 仕様書リンクなし、の **AND 条件** を満たす場合。

---

## Step 1a: 周辺チケット参照（コンテキスト補完）

チケット本体の内容では何を作ればよいか判断できない場合に実施する。
**調査は1回の参照で打ち切る。不明点は補完履歴ログに残して先へ進む。**

| 参照順序 | 対象 | 確認内容 |
|---------|------|---------|
| 1 | **親チケット / Epic** | ビジネス目的・全体スコープ・背景・上位 AC |
| 2 | **リンクされたチケット**（blocks / is blocked by） | 依存関係・前提条件・既実装の範囲 |

**チケット管理ツール別の取得方法:**

```bash
# Jira（jira-cli スキルを使用）
jira issue view EPIC-xxx      # 親 Epic 確認
jira issue view PARENT-xxx    # 親チケット確認

# GitHub Issues
gh issue view 123              # 親 issue または関連 issue 確認

# Linear
# mcp__claude_ai_Linear__get_issue を使用

# Notion / その他
# チケット内の URL から notion-fetch 等を使用
```

収集結果は出力ファイルの「コンテキスト収集ログ」セクションに記録してから Step 1 に戻る:

```markdown
## コンテキスト収集ログ
- **参照したチケット**: {チケットID} — {タイトル}
- **判明したコンテキスト**: {ビジネス目的・スコープ等}
- **未解決事項**: {それでも不明な点} → 補完履歴ログへ
```
