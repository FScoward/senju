# 業務フロー整合性チェック

AC を磨く前に、対象チケットが **業務プロセス全体の中で整合しているか** を確認する手順。
`impact-analysis.md`（コード波及）と補完関係にある。こちらは「業務ロジック・状態・ルール」の整合性を見る。

> **位置づけ**: refine-ticket の責務は AC 文書の品質。
> このステップで得た情報は AC の補完（特に異常系・前提条件・副作用）に反映する。
> Tier 1 はスキップ可。Tier 2/3 は必須。

---

## いつ実施するか

| 状況 | 実施する？ |
|------|---------|
| Tier 1（CRUD・設定変更・変更ファイル3未満） | スキップ可 |
| Tier 2/3 | **必須** |
| 既存機能の仕様変更・拡張 | **必須**（ビジネスルール競合リスクが高い） |
| 新規機能 | 推奨（状態遷移・ロールの整合性確認） |

---

## 4観点チェックリスト

### 観点A: エンティティ状態遷移の整合性

チケットが触るエンティティ（Order・User・Payment・Alert 等）の **状態遷移が有効か** を確認する。

**確認すること:**
- このチケットで追加・変更する状態遷移は、既存の遷移図と矛盾しないか
- 遷移元として有効な状態は何か（`Given` 条件に含めるべき前提）
- 遷移先として到達できない状態の操作が AC に含まれていないか
- 他チケット・既存実装で同じエンティティの別遷移が定義されていないか

**収集コマンド例:**
```bash
# enum の状態一覧を確認
rg -n "enum.*State|sealed.*class.*Status" --type=kt
# 遷移ロジックを確認
rg -n "\.status\s*=" --type=kt
```

**AC への反映:**
- `Given` に「エンティティが XX 状態である」という前提を明示する
- 無効な遷移（例: CANCELLED → SHIPPED）が AC に含まれていれば削除または異常系として追加する
- 他チケットと競合する遷移があれば補完履歴ログに記録して申し送る

---

### 観点B: 業務フロー前後工程との連携

このチケットの処理が **業務ステップとしての前後に何があるか** を確認する。

**確認すること:**
- このチケットの処理が「成立するために必要な前工程」はACの前提条件に含まれているか
- このチケットの処理結果を「後工程が受け取る」場合、その後工程の期待値はACに副作用として含まれているか
- 後工程が存在する場合、部分失敗（このチケット成功・後工程失敗）時の補償処理はACに含まれているか

**典型的な見落とし:**
| ドメイン | 前工程の見落とし | 後工程の見落とし |
|---------|---------------|---------------|
| 在庫更新 | 注文確定が前提 | 欠品時の発注トリガー・会計連携 |
| 決済処理 | カート確定が前提 | 配送手配・メール送信 |
| ユーザー作成 | メール認証完了が前提 | ウェルカム通知・権限セットアップ |
| アラート生成 | サーベイ回答締切が前提 | 管理者通知・ダッシュボード集計 |

**AC への反映:**
- 前工程の未完了ケースを異常系 AC として追加する
- 後工程への副作用（イベント発行・通知・DB更新）を副作用 AC として追加する
- Saga パターンで補償処理が必要な場合、`[補完]` マーク付きで AC に追加し補完履歴ログに記録する

---

### 観点C: ビジネスルールの一貫性・競合

同ドメインの **他チケット・既存実装で定義済みのビジネスルール** との矛盾がないか確認する。

**確認すること:**
- このチケットで定義・変更するビジネスルールは、他チケット（同 Epic・隣接 Epic）の AC と矛盾しないか
- 既存コードに実装されたルール（バリデーション・ガード条件）とこのチケットの AC が食い違っていないか
- ルールを変更するチケットの場合、そのルールに依存している既存 AC への影響が考慮されているか

**収集コマンド例:**
```bash
# 同ドメインの既存バリデーション・ビジネスルールを確認
rg -n "require|check|validate|must|illegal" --type=kt -- src/main/kotlin/
```

**AC への反映:**
- 矛盾するルールを発見した場合は補完履歴ログに「競合」として記録し、どちらが正しいかをユーザーに問い合わせるコメントを残す
- 既存ルールとの整合が取れれば、そのルールをACの `Given` または `Then` の具体値として反映する

---

### 観点D: 補償処理・ロールバックシナリオ

複数ステップにまたがる処理で **後続ステップが失敗した場合の補償** が AC に含まれているか確認する。

**確認すること:**
- このチケットの処理が成功した後に後続処理が失敗した場合、このチケットの結果を「戻す」必要があるか
- 複数テーブル・複数サービスにまたがる更新がある場合、トランザクション境界は明確か
- 補償処理が必要な場合、その AC は明示されているか

**適用判断基準（Saga 的な処理の兆候）:**
- チケット内に「〜した後に〜する」という複数ステップがある
- 外部サービス（メール・決済・通知）への副作用が含まれる
- 「成功した場合のみ後続処理を行う」という条件分岐がある

**AC への反映:**
- 補償が必要なケースを `[補完]` 付き異常系 AC として追加する
- トランザクション境界が曖昧な場合は、非機能要件 AC として「〜まで1トランザクションであること」を追加する

## 出力: 業務フロー整合性レポート

`refine-ticket` の出力ファイルに以下のセクションを追加する。
**該当なしの観点も「該当なし」と明記**する（沈黙は「見落とし」と区別がつかない）。

```markdown
## 業務フロー整合性レポート

### A. エンティティ状態遷移
- `Order.status`: PENDING → CONFIRMED → SHIPPED の遷移を確認。CANCELLED は CONFIRMED 以降のみ有効。
  → AC2 の Given に「Order が CONFIRMED 状態」を追加した

### B. 前後工程連携
- 前工程: 「注文確定」チケット (APP-xxx) の完了が前提。AC1 の Given に追加
- 後工程: 「会計連携バッチ」が在庫更新イベントを購読。副作用 AC として AC5 [補完] に追加

### C. ビジネスルール
- 「割引クーポンは1注文1回まで」ルールが APP-yyy の AC3 に定義済み。
  このチケットの AC4 と整合確認 → 矛盾なし

### D. 補償処理
- 在庫確保後に配送手配が失敗するシナリオあり。補償として在庫を戻す処理の AC [補完] を追加
```

## アンチパターン

| アンチパターン | 問題 | 改善 |
|---------------|------|------|
| 「業務フロー影響なし」とだけ書く | 4観点を見たか不明 | 観点ごとに「該当なし」を明記 |
| 正常系の状態遷移しか見ない | 無効遷移・競合遷移を見落とす | 遷移元の全状態パターンを確認 |
| 後工程への副作用を AC に書かない | テスターが副作用の検証漏れを起こす | イベント・通知・DB変化を副作用 AC に明示 |
| 補償処理を「別チケット」と言って先送りにする | Saga が壊れた状態で本番に入る | 補償が必要なら `[補完]` AC として明示し申し送る |

---

## 補完履歴ログとの関係

- 業務フロー整合性調査で **AC に推定・補完を入れた** 場合は、SKILL.md の「補完履歴ログ」にも記録する
- ビジネスルール競合が判明し **ユーザー確認が必要** な場合は、補完履歴ログに「競合・要確認」として記録する
