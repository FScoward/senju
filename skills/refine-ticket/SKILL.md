# refine-ticket

既に起票済みのチケットの **description と受け入れ条件（AC）を品質基準まで磨き上げる**スキル。
曖昧・抽象的・AC 不在のチケットを、第三者が実装・テストできる具体的な文書に変える。

> **核心原則: AC は「第三者が PASS/FAIL を機械的に判定できる」状態まで磨く。**
> 「正しく動く」「適切に処理する」では不十分。具体的なデータ値・状態・結果を伴う GWT+Examples で書く。
>
> **価値明文化原則: チケットには「誰のどんな体験が変わるか（ユーザー価値）」と「開発者にとっての技術的恩恵（開発者価値）」を明記する。**
> AC を書く前に価値が言語化されていると、「何のために磨くか」の軸が定まり、AC の過不足判断が正確になる。
>
> **品質原則: `references/ticket-quality-rules.md` の 8 セクション基準を満たした状態にする。**
> `check-ticket-quality.py` 等のスクリプトが存在する環境ではその通過も期待できるが、スクリプトへの依存はない。

## いつ使う / 使わない

**使う（既に起票済みチケットの文書を磨く）:**
- チケットの AC が抽象的・曖昧（「正しく表示される」等）で、実装/テストの基準にならない
- AC が無い、または1行しかないチケットを実装可能な状態にしたい
- チケットの description 本文を整理・書き直したい
- AC の過不足（異常系・境界値・副作用の抜け）をチェックして補完したい

**使わない（別スキルを使う）:**
| 状況 | 使うスキル |
|------|-----------|
| まだチケットになっていない要求を対話で引き出したい（未起票段階） | `software-requirements` |
| 初回プロンプトを即席で Goal / Constraints / AC に整える | `init-prompt` |
| チケットを複数の実装単位に**分割**したい | `kouunryuusui`（E3）/ `vertical-slice` 等 |

## 入力 / 出力契約

### 入力
| 引数 | 必須 | 説明 |
|------|------|------|
| チケット | ○ | チケット ID / URL / 内容テキストのいずれか |
| Tier | 任意 | 仕様投資レベル（1/2/3）。**渡されれば使う、無ければ自前判定**（下記） |
| 出力ファイル名 | 任意 | **渡された時のみファイル出力**（例: kouunryuusui は `spec-draft.md` を渡す）。**渡されない単独利用時はファイルを作らずチャットに直接出力する** |

### 出力先（呼び出し方で変わる）

> **重要: 出力ファイル名が渡されない単独利用時は、一時ファイルを生成しない。**
> 以下の 1〜6 を **assistant メッセージ（チャット）に直接出力**する。
> 出力ファイル名が**明示的に渡された場合のみ**（kouunryuusui 等の上位フローから呼ばれた時）、
> そのファイルに書き出す（呼び出し元がファイルを読んで次工程に使うため）。
> 単独利用でファイルに残したいとユーザーが明示した時だけ、保存先を確認してから書く。
>
> **注意（write-back）**: 単独利用時のチャット出力は実チケット（JIRA/GitHub/Linear）を自動更新しない。
> 出力された description/AC をユーザーが元チケットへコピーするか、各ツールの CLI/MCP で更新すること。
> 「品質原則」は kouunryuusui 連携のファイル出力経路でのみ完全に機能する。

出力する内容（出力先がファイルでもチャットでも同一）:
0. **根本問い直しレポート**（Step 2b の結果。XY問題・代替手段・放置コスト・スコープ妥当性の4観点。一時停止した場合はここで出力を終了する）
1. **磨かれた description**（ユーザー価値・開発者価値を先頭に置き、テンプレート全セクションを網羅した本文）
2. **AC**（GWT + Examples 形式、`ac-guidelines.md` 準拠）
3. **7項目品質レポート + hooks チェック通過見込み**（充足 / 補完した項目の一覧 + 各セクションの pass/warn 予測）
4. **影響範囲レポート**（コード波及 7観点の調査結果。`impact-analysis.md` 参照）
5. **業務フロー整合性レポート**（状態遷移 / 前後工程 / ビジネスルール競合 / 補償処理 の 4観点。Tier 2/3 のみ。`business-flow-analysis.md` 参照）
6. **補完履歴ログ**（`[補完]` で追加・推定した内容。下記「補完履歴ログ」参照）
7. **優先度判定**（P0〜P3 + スコア根拠。下記「優先度判定仕様」参照）

## Tier 判定 / 価値判定（統合）

AC の詳細度は Tier で決まる。Tier は**チケットの価値・優先度スコアから導出する**。

`refine-ticket` は **2 通りで Tier を得る**:

1. **呼び出し元が Tier を渡した場合** → その Tier をそのまま使う（kouunryuusui 等から呼ばれる時）
2. **渡されない（単独利用）場合** → 4軸スコアリングで自前判定する（下記「優先度判定仕様」参照）:

| スコア | 優先度 | Tier | AC 詳細度 |
|---|---|---|---|
| 10〜12 | **P0 Critical** | **Tier 3** | フル（正常系 + 境界値 + 異常系 + 副作用 + 非機能） |
| 7〜9   | **P1 High**     | **Tier 3** | フル |
| 4〜6   | **P2 Medium**   | **Tier 2** | 標準（正常系 + 主要異常系） |
| 0〜3   | **P3 Low**      | **Tier 1** | 最小（正常系のみ） |
| —      | **実施不推奨**  | **Tier 0** | refine を中断。やらない根拠を提示して終了する |

**Tier 0 判定条件**: 4軸スコアが全て 0 点、または「ビジネス緊急度 0 かつ技術リスク 0」で価値を見出せない場合。最終判断は人間（PM/PO）が行う。

判定結果と根拠は出力冒頭に記録する。

## フロー

```

├─ 0. 記憶読み込み       → ~/.claude/skills-memory/refine-ticket/memory.md
│     ファイルが存在すれば、チケットのドメイン領域を推定して一致する
│     エントリ（ドメイン制約・誤補完履歴）を抽出し Step 8 に注入する
│     ## Recurring Review Findings セクションが存在すれば、一致ドメインの
│     エントリを抽出して Step 7（過不足分析）のヒントとして注入する
├─ 1. 存在チェック       → references/peripheral-context.md（Step 1）
│     チケットに AC があるか / 仕様書リンクがあるか / 内容が十分か
├─ 2. コンテキスト補完   → references/peripheral-context.md（Step 1a）
│     内容不十分なら親チケット/Epic を1回だけ参照
├─ 2a. 価値明文化
│     ユーザー価値: 対象ユーザー（ロール/ペルソナ）・体験の変化・Why Now を整理
│     開発者価値: 技術的改善点（保守性・テスト容易性・負債解消等）・将来変更への影響を整理
│     元チケットに記載がない場合は [補完] マークを付けて推定し、補完履歴ログに記録する
├─ 2b. 根本問い直し（Problem Reframing）
│     チケットの方向性そのものを問い直す（AC を磨く前に実施）:
│     ① XY問題チェック: 「〜を実装する」形で書かれている場合、背後の問題（Why）を
│       言語化できるか？解決策が先に決まっていないか？
│     ② 代替手段: 同じ問題を解く別アプローチを最大3案列挙し、
│       比較軸（コスト・リスク・速度・保守性）で評価する
│     ③ 放置コスト: このチケットをやらない場合の影響・リスクを明示する
│     ④ スコープ妥当性: 過剰実装（YAGNI違反）・過小実装（後続チケットの量産）になっていないか
│     → 以下の条件に当てはまる場合は refine を一時停止し、
│       問い直し結果をユーザーに提示して方向確認を求める:
│       - XY問題の可能性が高い（背後の問題と実装内容がズレている）
│       - 代替手段の評価で「現チケットの方針が最善でない」と判断できる
│       - スコープが明らかに過剰または分割すべきと判断できる
│     → 問題なしと判断した場合は根本問い直しレポートに「方針確認: OK」と記録して続行する
├─ 3. 価値・優先度判定（統合）  → 上記「Tier 判定 / 価値判定」
│     4軸（ユーザー影響度 / ビジネス緊急度 / 技術リスク / 依存関係）をスコアリング
│     スコアから P0〜P3 と Tier を同時に導出する
│     Tier 0 と判定した場合は refine を中断し、やらない根拠を提示して終了する
├─ 4. 影響範囲洗い出し   → references/impact-analysis.md
│     呼び出し元 / 呼び出し先 / データフロー / 契約 / 既存テスト
│     / 横断的関心事 / リリース順序 の 7観点で grep ベース調査
│     既存修正時は必須。Tier 1 と新規開発は軽量版
├─ 5. 業務フロー整合性   → references/business-flow-analysis.md
│     エンティティ状態遷移 / 前後工程連携 / ビジネスルール競合
│     / 補償処理 の 4観点を確認
│     Tier 1 はスキップ可。Tier 2/3 は必須
├─ 5b. hooks チェックセクション補完  → 下記「hooks チェックセクション仕様」
│     テンプレートの全セクションを埋める（下記を参照）:
│     デザイン / 影響範囲（テンプレート記載用） / タスク / テスト観点 / リリース計画 / 成功指標（KPI）
│     記載がない項目は [補完] マークを付けて推定し、補完履歴ログに記録する

│     7a. Tier 2/3 の場合: Skill(test-matrix) を呼び出し、Step 6 で整えた AC 群をテストマトリクスに変換する
```

- **`references/peripheral-context.md`** — 存在チェック・周辺チケット参照
- **`references/impact-analysis.md`** — 影響範囲（コード波及）7観点チェックリスト・影響範囲レポート出力テンプレート
- **`references/business-flow-analysis.md`** — 業務フロー整合性 4観点（状態遷移・前後工程・ビジネスルール競合・補償処理）・業務フロー整合性レポート出力テンプレート
- **`references/ac-completion.md`** — フォーマット修正・仕様書→US+AC 生成・出力テンプレート

## 補完履歴ログ

AC を補完・推定・解釈・分割したら、その履歴を残す。**何を自分で埋めたかを後から追えるようにする**ため。

記録対象:
- `[補完]` マーク付き AC を追加した（仕様書・コード・ドメイン知識からの推定）
- チケット本文に書かれていない前提・制約・非機能要件を追加した
- 親チケット参照で得た情報を仕様として採用した
- AC を分割・統合・順序変更した

出力ファイルの末尾に記録する:

```markdown
## 補完履歴ログ
- **[補完] AC3（異常系）**: 元 AC に異常系が無かったため、UserEntity のバリデーション
  ルール（name: not blank）から推定して追加。根拠: コードベース。
- **前提追加**: 期間フィルタの上限 90 日。元チケットに記載なし。性能観点で必須と判断。根拠: ドメイン知識。
```

> **責務分担**: `refine-ticket` は補完履歴ログを**出力するだけ**。Decision Record 化はしない。
> `kouunryuusui` 等の上位フローから呼ばれた場合、**呼び出し元がこの履歴を読んで DR 化**する。
> 単独利用時は履歴ログがそのまま「自分が何を埋めたか」の記録になる。

## チケット品質ルール（セクション定義）

詳細は `references/ticket-quality-rules.md` を参照。
Step 5b で 8 セクション（課題・背景 / A.C. / デザイン / 影響範囲 / タスク / テスト観点 / リリース計画 / 成功指標）をすべて補完し、
品質チェック出力テンプレートを Step 10 の品質レポートに含める。

## 優先度判定仕様

Step 5a で 4軸をスコアリングし、P0〜P3 を判定する。

### スコアリング軸（各 0〜3 点）

| 軸 | 0点 | 1点 | 2点 | 3点 |
|----|-----|-----|-----|-----|
| **ユーザー影響度** | 影響なし | 一部ユーザーに軽微 | 多くのユーザーに中程度 | 全ユーザーに重大 or データ損失リスク |
| **ビジネス緊急度** | いつでもよい | 次の四半期までに欲しい | 次のリリースまでに必要 | 今すぐ対応が必要（障害・法的義務等） |
| **技術リスク** | 放置しても無害 | 軽微な負債蓄積 | 将来の変更を阻害 | セキュリティ脆弱性・システム不安定化 |
| **依存関係** | 他チケットに依存なし | 1チケットが待ち状態 | 複数チケットがブロック中 | リリース全体がブロック中 |

### 優先度マッピング（合計点 → P0〜P3）

| 合計点 | 優先度 | 意味 |
|--------|--------|------|
| 10〜12 | **P0 Critical** | 即時対応。スプリントを中断してでも着手 |
| 7〜9   | **P1 High**     | 次スプリントに必ず入れる |
| 4〜6   | **P2 Medium**   | バックログに積んで計画的に対処 |
| 0〜3   | **P3 Low**      | 余裕があるときに対応 |

### 出力フォーマット

```markdown
## 優先度判定

**判定結果**: P1 High（スコア: 8/12）

| 軸 | スコア | 根拠 |
|----|--------|------|
| ユーザー影響度 | 2 | ログイン済みユーザー全員が対象機能を使用する |
| ビジネス緊急度 | 3 | v2.0 リリース（来月）のブロッカー |
| 技術リスク | 2 | 現状の回避策が技術負債化しており後続実装を阻害 |
| 依存関係 | 1 | APP-456（通知機能）が本チケット完了待ち |

**推奨アクション**: 次スプリントのトップに積む。APP-456 との順序を PM と確認すること。
```

### 注意事項

- スコアリング根拠はチケット本文・影響範囲レポート（Step 4）・業務フロー整合性レポート（Step 5）から引く
- 元チケットに優先度が既に設定されている場合は、判定結果と比較してズレがあれば `[要確認]` マークを付けて提示する
- 優先度の最終決定は人間（PM / PO）が行う。本スキルの判定はあくまで**提案**

## 価値セクション仕様

Step 2a で明文化した価値は、磨かれた description の先頭セクションとして出力する。

```markdown
## ユーザー価値
- **対象ユーザー**: <ロール・ペルソナ（例: 一般会員、管理者、外部連携システム）>
- **体験の変化**: <この機能がないときとあるときで何が変わるか・1〜2行>
- **Why Now**: <なぜこのタイミングで必要か（優先度の根拠）>

## 開発者価値
- **技術的改善**: <コード品質・保守性・テスト容易性・技術的負債解消などへの寄与>
- **将来への影響**: <この変更が後続の開発・拡張をどう楽にするか>
```

### 記述ガイドライン

**ユーザー価値**:
- 「ユーザーが便利になる」は不可。誰が・何が変わるかを具体的に書く
- Why Now は省略可だが、「技術的負債解消」「リリース前提」等の場合は明記する

**開発者価値**:
- 純粋なビジネス機能チケットでも、「テスタビリティ向上」「既存コードの制約緩和」等の副次的恩恵を積極的に見つけて記述する
- 技術的恩恵が全くないチケットは「N/A（ビジネス機能のみ）」と明記してよい

**補完ルール**:
- 元チケットに価値の記述がない場合、コンテキスト・コードベース・ドメイン知識から推定して `[補完]` マークを付ける
- 推定した価値は補完履歴ログに記録する

## 使い方

```
# 既存チケットの AC を磨く（単独利用）
/refine-ticket APP-1234

# 内容テキストを直接渡す
/refine-ticket "ユーザーが一覧をフィルタできる。AC: 正しく絞り込まれる"

# 上位フロー（kouunryuusui 等）から呼ばれる場合は Tier と出力ファイルを受け取る
```

## 記憶（クロスセッション学習）

### 読み込みタイミング（Step 1 着手前）

`~/.claude/skills-memory/refine-ticket/memory.md` が存在すれば読む。

1. チケットのドメイン領域を推定する（認証・検索・アラート・課金・通知 等）
2. memory.md のエントリから `domain:` が一致するものだけを抽出する
3. 抽出した **ドメイン制約** を Step 8（自動補完）に注入する
   → 同じ制約を再推定しない、矛盾した補完を避ける
4. 抽出した **誤補完履歴** も Step 8 に注入する
   → 過去に否定された補完を繰り返さない

ドメインが不明瞭なときは全エントリを読む（エントリが少ないうちは問題なし）。

### 書き込みタイミング（Step 10 出力後）

`~/.claude/skills-memory/refine-ticket/memory.md` に追記する。
**書くのはドメイン知識ベースの `[補完]` のみ**（コードベース・仕様書由来は不要）:

```markdown
### YYYY-MM-DD — <チケットID>
- **domain**: <認証|検索|アラート|課金|通知|etc.>
- **推定ドメイン制約** (根拠=ドメイン知識の [補完] のみ):
  - 例: 「期間フィルタ上限90日（パフォーマンス観点で推定）」
- **誤補完** (ユーザー/downstream に否定された [補完]):
  - （なければ省略）
```

#### 誤補完の後追い記録

誤補完の訂正は refine 実行直後でなく **後のセッションで発覚することがある**。
ユーザーが「あの補完は間違いだった」と指摘したら、該当エントリを探して `誤補完` 欄に追記する。

### Recurring Review Findings（review-loop-duo フィードバック）

`## Recurring Review Findings` セクションが存在する場合:
1. チケットのドメイン領域に一致するエントリ（`domain:` が一致）を抽出する
2. Step 7（過不足分析）に注入する: 「レビューで繰り返し検出されたパターン → AC にこの観点が含まれているか確認」
3. Step 8（自動補完）では使わない — あくまで人間が判断する AC ヒントとして提示するのみ

エントリのスキーマ（`review-loop-duo reflect` が提案し、人間が承認して追記）:

```markdown
### <domain>:<pattern-slug>
- **domain**: <認証|権限|通知|課金|etc.>
- **ac_hint**: AC に含めるべきチェック観点（前向き表現。1〜2行）
  例: 「参照系と更新系エンドポイントの権限が対称か AC に明記する」
- **source_signal**: review-loop-duo の lens signature（参照元）
- **count**: 検出回数（参考値）
```

書き込み: `review-loop-duo reflect` が lens 信号を昇格させる際に追記候補を提案する。`refine-ticket` 自身は書き込まない。
