# レビュー観点（3並列エージェントの詳細）

mihari は 3 つの並列エージェント（X / Y / Z）でテスト充足性をレビューする。各エージェントは担当観点を持ち、重複せず網羅性を担保する。

---

## Agent X: 等価分割 + 境界値分析

### 担当観点

**入力値のクラス分けと境界の検証**を担当する。

### チェック項目

| チェック | 内容 | 判定基準 |
|---------|------|---------|
| 等価分割 | AC の各入力に対して、有効クラス/無効クラスから代表値を1つ以上テストしているか | 有効クラスの代表値が未テスト → Critical。無効クラスが未テスト → Critical |
| 下限境界 | 数値・文字列長・コレクションサイズ等の最小値をテストしているか | min-1 / min / min+1 のいずれかが欠けている → Warning（両端必須：Critical） |
| 上限境界 | 上限値をテストしているか | max-1 / max / max+1 が欠けている → Warning（両端必須：Critical） |
| null / 空 | null、空文字、空配列、空オブジェクトをテストしているか（nullableな入力のみ） | null 許容で未テスト → Critical |
| 数値の特殊値 | 0、-1、Integer.MAX_VALUE、小数点ゼロ等 | 数値入力で未テスト → Warning |
| 文字エンコーディング境界 | 絵文字・全角・サロゲートペアを含む文字列 | 文字列入力で国際化要件あり時に未テスト → Warning |

### 判定出力フォーマット

```markdown
## Agent X: 等価分割 + 境界値分析 結果

### 指摘サマリー
- Critical: {N}件
- Warning: {M}件

### 詳細

#### Critical: AC1 のメールアドレス無効クラス未テスト
- 該当 AC: AC1（ユーザー登録）
- 不足: `not-an-email`, `@.com`, 空文字 のいずれも未テスト
- 推奨テスト: `registerWithInvalidEmailFormat("not-an-email")`
- 出典観点: 等価分割

#### Warning: AC2 の氏名最大長境界未テスト
- 該当 AC: AC2（氏名登録）
- 不足: 最大長255文字のちょうど境界テストが存在しない
- 推奨テスト: `registerWithMaxLengthName(length=255)`, `registerExceedingMaxLength(length=256)`
- 出典観点: 境界値分析（max / max+1）
```

### プロンプトテンプレート（エージェント起動時）

```
あなたは「テストレビュアー Agent X（等価分割・境界値担当）」です。
以下のテストコードを、等価分割と境界値分析の観点で網羅性をレビューしてください。

【入力】
- AC: {AC本文}
- 実装ファイル: {実装コードの抜粋}
- テストファイル: {テストコードの抜粋}

【チェック観点（これ以外は見ない）】
1. 有効クラス・無効クラスの代表値が各 AC でテストされているか
2. 数値/文字列長/コレクションサイズの min-1/min/min+1/max-1/max/max+1 境界
3. null / 空 の網羅
4. 数値の特殊値（0、-1、MAX_VALUE 等）
5. 文字エンコーディング境界（絵文字・全角・サロゲート）

【出力】
review-patterns.md の「判定出力フォーマット」に従い、Critical / Warning を列挙する。
assertionの品質やコード品質・ドメインロジック等、担当観点外には触れないこと（Agent Z/Y が担当）。

【過検出回避】
以下は過検出なので指摘しないこと（前回ラウンドで誤検出されたパターン）:
{キャリブレーション情報（イテレーション2以降で注入）}
```

## Agent Y: デシジョンテーブル + 状態遷移

### 担当観点

**複数条件の組合せと状態の遷移**を担当する。

### チェック項目

| チェック | 内容 | 判定基準 |
|---------|------|---------|
| デシジョンテーブル網羅 | 条件 A/B/C の真偽組合せ全パターンがテストされているか（最低限必要な組合せ） | 必要組合せの未テスト → Critical |
| 条件分岐の独立性 | 1テスト内で複数条件を混ぜず、条件ごとに独立テストしているか | 混ざっている → Warning |
| 正常状態遷移 | 状態遷移図の有効パスが全てテストされているか | 有効パス未テスト → Critical |
| 不正状態遷移 | 無効な状態遷移が拒否されることをテストしているか（例: CLOSED → IN_PROGRESS） | 無効パス拒否未テスト → Critical |
| 冪等性 | 同じ操作を2回実行して結果が変わらないことをテストしているか（冪等性が AC にある場合） | 冪等性AC ありで未テスト → Warning |
| 権限・認可の条件マトリクス | ログイン状態 × 権限の全組合せ（ログイン×権限あり/なし、未ログイン等） | 認可要件あり かつ 未テスト → Critical |

### 判定出力フォーマット

```markdown
## Agent Y: デシジョンテーブル + 状態遷移 結果

### 指摘サマリー
- Critical: {N}件
- Warning: {M}件

### 詳細

#### Critical: 認可マトリクスの未テスト組合せ
- 該当 AC: AC3（権限制御）
- 不足テストパターン:
  | ログイン | 権限 | 期待 | テスト有無 |
  |---------|------|------|----------|
  | ✅ | ✅ | 200 | ✅ |
  | ✅ | ❌ | 403 | ❌（未テスト） |
  | ❌ | ✅ | 401 | ❌（未テスト） |
  | ❌ | ❌ | 401 | ✅ |
- 推奨テスト: `accessDeniedWhenLoggedInWithoutPermission()`, `unauthorizedWhenNotLoggedIn()`
- 出典観点: デシジョンテーブル
```

### プロンプトテンプレート

```
あなたは「テストレビュアー Agent Y（デシジョンテーブル・状態遷移担当）」です。
以下のテストコードを、条件組合せと状態遷移の観点で網羅性をレビューしてください。

【入力】
- AC: {AC本文}
- 実装ファイル: {実装コードの抜粋}
- テストファイル: {テストコードの抜粋}

【チェック観点（これ以外は見ない）】
1. 複数条件の組合せ（A×B×C）のうち意味のあるパターンが網羅されているか
2. 状態遷移の有効パス・無効パス両方がテストされているか
3. 冪等性（AC にある場合）
4. 認可/権限の条件マトリクス

【出力】
review-patterns.md の「判定出力フォーマット」に従い、Critical / Warning を列挙する。
等価分割・境界値・エラー推測等、担当観点外には触れないこと。

【過検出回避】
{キャリブレーション情報}
```

## Agent Z: エラー推測 + 副作用 + assertion 品質

### 担当観点

**壊れやすい箇所・副作用・アサーションの質**を担当する。

### チェック項目

| チェック | 内容 | 判定基準 |
|---------|------|---------|
| 並行性・競合 | 同時実行・二重送信・楽観ロック競合のテスト | 並行処理の AC あり かつ 未テスト → Critical |
| 外部依存の失敗 | API タイムアウト、500 エラー、DB 切断、接続プール枯渇 | 外部依存あり かつ 未テスト → Warning |
| タイムゾーン境界 | UTC midnight、DST 切替、月末/年末 | 時刻依存 AC で未テスト → Warning |
| 副作用の検証 | DB 更新、イベント発行、通知送信、ログ出力 | 副作用あり かつ assertion なし → Critical |
| 戻り値の検証 | 戻り値が期待通りか（状態だけでなく） | assertion が副作用のみ → Warning |
| assertion の具体性 | `assertNotNull` のみでなく具体値検証 | 具体値未検証 → Warning（Critical if 唯一の assertion） |
| assertion の多角性 | 1テストに意味のある assertion が2つ以上（戻り値 + 副作用、状態 + イベント等） | 1 assertion のみ → Warning |
| 空テスト | `assertTrue(true)`、`// TODO` 等 | 検出 → Critical（即 T3 差し戻し） |
| エラーメッセージの検証 | 例外発生時のメッセージ・エラーコード | 例外テストで message/code 未検証 → Warning |

### 判定出力フォーマット

```markdown
## Agent Z: エラー推測 + 副作用 + assertion 品質 結果

### 指摘サマリー
- Critical: {N}件
- Warning: {M}件

### 詳細

#### Critical: 副作用（ウェルカムメール送信）の assertion 欠落
- 該当テスト: RegisterUserUseCaseTest#success
- 不足: 登録成功テストで戻り値は検証しているが、ウェルカムメール送信（副作用）の検証がない
- 推奨: `verify(emailService).sendWelcome(eq(user.email))` を追加
- 出典観点: 副作用検証

#### Warning: assertion が1つのみ
- 該当テスト: UpdateUserUseCaseTest#success
- 現状: `assertEquals(updatedName, result.name)` のみ
- 推奨追加: updatedAt が更新されていること、変更イベントが発行されていることの検証
- 出典観点: assertion 多角性
```

### プロンプトテンプレート

```
あなたは「テストレビュアー Agent Z（エラー推測・副作用・assertion品質担当）」です。
以下のテストコードを、壊れやすさと検証の質の観点でレビューしてください。

【入力】
- AC: {AC本文}
- 実装ファイル: {実装コードの抜粋}
- テストファイル: {テストコードの抜粋}

【チェック観点（これ以外は見ない）】
1. 並行性・競合・二重送信
2. 外部依存の失敗（タイムアウト・500・切断）
3. タイムゾーン・時刻境界
4. 副作用（DB/イベント/通知）の assertion
5. 戻り値 assertion の有無
6. assertion の具体性（assertNotNull だけでないか）
7. 空テスト・TODO テストの検出
8. 例外テストでの message/code 検証

【出力】
review-patterns.md の「判定出力フォーマット」に従う。
等価分割・境界値・状態遷移は担当観点外。

【過検出回避】
{キャリブレーション情報}
```

## エージェント起動パラメータ（呼び出し時）

| パラメータ | 値 |
|-----------|-----|
| subagent_type | `general-purpose` |
| model | `sonnet` |
| 並列度 | 3（X/Y/Z 同時起動） |
| 出力形式 | 上記「判定出力フォーマット」で統一 |

opus への昇格条件: 3 Round 経過して同一カテゴリ指摘が連続する場合、次 Round の該当 Agent を `opus` に切り替えて根本原因を分析する（loop-protocol.md 参照）。
