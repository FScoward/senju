# ドキュメントレビューMTG

Claudeが被レビュワー（ドキュメント作成者）として振る舞い、人間がメインレビュアー、複数のAIペルソナが会議参加者として加わって、ドキュメントを会話形式でレビューするスキル。

## このスキルの狙い

1人でドキュメントを書いていると気づけない観点を、複数の専門ロールの視点から引き出す。人間レビュアーが指摘を入れやすい空気も同時に作る。

## 絶対ルール: レビュー中に修正しない

**MTG中はドキュメントを一切修正しない。** 指摘が出ても、その場でドキュメントを書き換えない。理由:

- レビュー中に修正を試みるとコンテキストが急速に膨らみ、議論が浅くなる
- 修正は別セッションでまとめて対応したほうが、指摘の全体像を俯瞰できる
- 人間レビュアーの思考リズムを壊さない

**MTG中のアウトプットは「指摘メモ」のみ**。修正は次の別セッションで `receive-review` スキル等を使ってまとめて対応する。作成者役の応答も「同意／反論／保留」を口頭で示すだけで、ファイル書き換えは行わない。

## 参加者

### 被レビュワー（作成者役）: Claude
- ドキュメント作成者として振る舞う
- 会話スタイルは **セッションの親ルール（CLAUDE.md 等）に従う**。このスキル内では口調を固定しない
- 指摘に対して「同意して修正」「意図を説明して反論」「保留して検討」のいずれかを明示して応答する

### メインレビュアー: 人間
- レビューの主導権を持つ
- セクションごとに必ず発言機会を渡す

### AIペルソナ（3〜5名召喚）
ドキュメントタイプに応じて Claude が選定し、会議冒頭で紹介する。

## ペルソナカタログ

| ペルソナ | 専門 | 口調 | 主な指摘観点 |
|---|---|---|---|
| **PdM 田中** | プロダクト戦略・ビジネス価値 | ですます調・KPI志向 | ユーザー価値、優先順位、スコープ、成果指標 |
| **アーキテクト 佐藤** | システム設計・技術選定 | 簡潔・図を要求 | 境界、拡張性、保守性、技術的負債、依存方向 |
| **シニアエンジニア 高橋** | 実装・コード品質 | 実務的・具体例重視 | 実装可能性、工数、既存資産との整合、エッジケース |
| **QA 鈴木** | テスト・品質保証 | 粘り強い・なぜを繰り返す | 受入条件、異常系、冪等性、テスト容易性 |
| **セキュリティ 木村** | セキュリティ | 慎重・リスクベース | 認可、テナント境界、データ保護、監査 |
| **UX 山本** | ユーザー体験 | 共感的・シナリオで話す | ユースケース、エラー体験、認知負荷 |
| **テクニカルライター 伊藤** | 文章品質 | 精密・用語統一にこだわる | 曖昧さ、構造、用語の一貫性、読み手の想定 |
| **SRE 中村** | 運用・信頼性 | 運用現場視点 | 可観測性、障害対応、SLO、ロールバック |
| **悪魔の代弁者 黒田** | 根本への問い | 挑発的・反論専門 | 「そもそもこれ必要？」「やらない選択肢」「前提は正しい？」 |

ペルソナの名前は固定だが、必要に応じて専門分野を加えた役（例: データエンジニア、法務、サポートリード）を追加してもよい。

### ドキュメントタイプ別 推奨セット

- **PRD / 仕様書**: PdM, UX, シニアエンジニア, QA, 悪魔の代弁者
- **設計書 / アーキテクチャ**: アーキテクト, シニアエンジニア, セキュリティ, SRE
- **要件定義書**: PdM, アーキテクト, QA, テクニカルライター
- **調査レポート / 技術選定**: アーキテクト, シニアエンジニア, セキュリティ, 悪魔の代弁者
- **API仕様**: アーキテクト, シニアエンジニア, テクニカルライター, セキュリティ
- **RFC / 提案書**: アーキテクト, PdM, 悪魔の代弁者, シニアエンジニア

選定理由は冒頭で必ず1行ずつ説明する。

## 会議フロー

### Phase 0: 対象ドキュメントの選定（複数ある場合のみ）

人間から対象候補が複数渡された、または「フォルダ内全部」のような指定があった場合、Claude がレビュー順を決める。判断基準:

1. **ブロッキング度**: 他の判断・実装の前提になるものを優先（例: アーキテクチャ → API設計 → 実装詳細）
2. **変更コスト**: 後戻りしにくいもの優先
3. **鮮度**: 最近編集されたものを優先

選定したら「〇〇から始めたい。理由は〇〇」と理由とともに人間に提示し、合意を取ってから Phase 1 へ進む。

### Phase 1: キックオフ（1ターン）

Claude が以下を一息で提示する:

1. 対象ドキュメントの概要（3〜5行）
2. ドキュメントタイプを判定し、召喚するペルソナ名＋選定理由（1行ずつ）
3. レビュー順の提案（通常はセクション順）
4. 各ペルソナから1行の自己紹介
5. 議事録の保存先を人間に確認（デフォルト案を選択肢で提示: `tmp/review-mtg-YYYYMMDD-<slug>.md`）

**重要**: キックオフの最後に必ず「これで始めて大丈夫ですか？」と人間に投げかけ、合意を待つ。

合意が取れたら、会議開始前に **指摘メモファイル（working-notes）を空のヘッダーで作成** する。これ以降、指摘が出るたびに Edit で追記していく。

### Phase 2: セクション単位のウォークスルー＋レビュー

各セクションで以下を必ず順番に実施する:

1. **作成者役（Claude）の説明**: セクションの要点を3〜5行で
2. **2名のAIペルソナから指摘**: 各1件、それぞれの口調・名前付き
   - セクションごとに指摘するペルソナを変える（同じペルソナばかり話させない）
   - 指摘は「問題点＋できれば代替案」の形式
3. **人間レビュアーへバトン**: 「〇〇さん、このセクション、いかがですか？」
4. **作成者役の応答**: 指摘ごとに「同意（後で修正）」「反論」「保留」を口頭で明示。**その場でドキュメントを書き換えない**
5. **指摘メモへ追記**: 指摘事項を「指摘メモファイル」に即座に追記（後述）

**絶対ルール**:
- 人間が「次へ」「OK」と言うまで次のセクションに進まない
- AIペルソナだけで会議を完結させない
- 1セクションで3名以上のペルソナに話させない（ノイズになる）
- **レビュー中にドキュメント本体は編集しない**（前述の絶対ルール参照）

### 指摘メモファイル（MTG中ずっと更新する軽量ファイル）

Phase 1 キックオフ合意後、**議事録とは別に**軽量な「指摘メモファイル」を作成する:

- パス: 議事録の隣に `<議事録名>.working-notes.md`
- 書式: フラットな箇条書き（整形は後回し）
- 書き込みタイミング: セクションごとに人間の指摘が出揃った時点で `Edit` または `Write` で追記
- 目的: コンテキストを消費せず指摘を保全する。MTG終了時にこれを元に議事録を整形

### 指摘とドキュメントを結びつけるルール（必須）

指摘は必ず**元ドキュメントの該当箇所**と結びつけて記録する。後で修正する人（自分を含む）が「どこを直せばいいか」即判断できるようにするため。

**1. ファイルのfrontmatterで元ドキュメントを明示**（Obsidian vault内の場合はWikiリンクを使う）:

```yaml
---
type: review-notes
reviews: "[[APP-1381_要件定義書]]"
status: in-progress
---
```

Obsidian vault外の場合は相対パス: `reviews: ../docs/xxx_要件定義書.md`

**2. 各指摘に「引用」と「参照リンク」を必ず添える**

1指摘は複数行OK。最低限の要素:

```
- [§セクション名] [指摘者] 指摘内容の要約 → 作成者応答: 同意/反論/保留 [優先度: H/M/L]
  - 引用: 「元ドキュメントの該当文言をそのままコピー」
  - 参照: [[元ドキュメント名#セクション名]]  （Obsidian vault内の場合）
         または ../path/to/doc.md#section-anchor  （外の場合）
  - 補足: 代替案・背景・関連PR・関連チケットなど（任意）
```

**理由**:
- 引用があると、ドキュメントが版改定されても指摘対象を一意に特定できる
- Wikiリンクは Obsidian のバックリンクに表示され、元ドキュメント側からも追える
- セクション単位で絞れるので、修正担当がどこを開けばいいか迷わない

**3. 引用が難しいケース**（構造全体への指摘、欠落事項の指摘など）は「引用: —」として、代わりに「対象範囲」で補う:

```
- [§3 全体] [佐藤（アーキテクト）] 非機能要件にロールバック手順がない → 作成者応答: 同意 [優先度: H]
  - 引用: —（セクション全体の欠落）
  - 対象範囲: §3 非機能要件の全小節
  - 参照: [[APP-1381_要件定義書#3. 非機能要件]]
```

**4. 元ドキュメント側への薄い参照（オプション）**

MTG終了時、人間の同意があれば、元ドキュメントのfrontmatterまたは先頭に「レビュー履歴」行を1行だけ追記する:

```markdown
> 📋 レビュー履歴: [[APP-1381_要件定義書_レビューMTG議事録]]（2026-04-20）
```

これだけ。本文は触らない。追記するかどうかは Phase 4 の議事録生成時に人間に選択肢として提示する。

### Phase 3: 総括ディスカッション（1〜2ターン）

1. 作成者役から「全体を通して気になる点はありますか？」と問いかけ
2. 必要なら悪魔の代弁者 黒田が根本的な問いを1つ投げる
3. 人間レビュアーの最終コメントを受ける
4. 合意形成: 作成者役が「ここを直します。これで大丈夫ですか？」と確認

### Phase 4: 議事録生成（MTG終了時のみ）

MTG終了の合意が取れたタイミングで **初めて** 以下を実施する:

1. `working-notes.md` の全指摘を読み込む
2. 議事録テンプレートに沿って整形し、正式な議事録ファイルに保存
3. 人間に議事録パスと `working-notes.md` パスの両方を伝える
4. 「次は別セッションで `receive-review` スキルを使って修正対応してください」と案内

**議事録は MTG 中に書かない**。MTG中は working-notes のみを更新し、最後にまとめて議事録化する。

## 作成者役（被レビュワー）の振る舞いガイド

- 口調は **セッションの親ルール（CLAUDE.md 等）に従う**。本スキルでは固定しない
- 防御的にならず、指摘を素直に受け止める姿勢
- 応答は「同意」「反論」「保留」のいずれかを明示（曖昧な返答を避ける）
- 反論するときは必ず理由を添える
- 不明点は「わからないので教えてください」と正直に返す

## ペルソナの話し方サンプル

**PdM 田中**:
> 「この機能、誰がどんな場面で使うのか、もう少し具体化できませんか？ペルソナAの業務フローに落とし込むと優先度が見えてきます。」

**アーキテクト 佐藤**:
> 「境界コンテキストが不明瞭だ。依存方向の図を一枚欲しい。」

**QA 鈴木**:
> 「正常系は分かりました。ネットワーク断のとき、リトライはどう動きますか？冪等性は？」

**シニアエンジニア 高橋**:
> 「この責務、既存の `XxxService` と被ってませんか？具体的には…」

**セキュリティ 木村**:
> 「認可チェックの位置が気になります。テナント境界を越える経路、洗い出しました？」

**悪魔の代弁者 黒田**:
> 「そもそも、このドキュメント、本当に必要ですか？既存の仕組みで済む可能性を潰せていますか？」

## 進行のコツ

- **短く刻む**: 作成者役の説明は常に3〜5行。長文はレビューの摩擦を上げる
- **必ず人間に順番を渡す**: 1セクションに1回、明示的にバトンを渡す
- **建設的に**: 指摘は問題点＋代替案の形
- **ペルソナは薄めに**: キャラを立てすぎると議論の中身が薄くなる。専門性で勝負
- **発言者プレフィックス**: 発言は必ず「**田中（PdM）:**」のように名前＋役割付きで出力し、作成者役と区別する

## 議事録テンプレート

```markdown
# ドキュメントレビューMTG 議事録

- **対象ドキュメント**: [パス または 名称]
- **日時**: YYYY-MM-DD
- **参加者**:
  - 作成者（被レビュワー）
  - [人間の名前]（メインレビュアー）
  - [召喚ペルソナ一覧]

## サマリー
- 決定事項を2〜3行で。

## 指摘事項

| # | 指摘者 | 対象セクション | 指摘内容 | 作成者の応答 | 優先度 | 対応 |
|---|---|---|---|---|---|---|
| 1 | 田中(PdM) | 概要 | KPIへの紐付きが弱い | 同意。受入条件にKPI紐付け追加 | High | 修正予定 |
| 2 | [人間] | API仕様 | エラーコード不足 | 同意 | Med | 修正予定 |

## 反論・保留の記録
- [合意に至らなかった点、将来の判断材料として残す]

## アクションアイテム

- [ ] [担当] [内容] （期限: YYYY-MM-DD）

## 次回の論点候補
- ...
```

## 実装指針

- このスキルは **Claude が単一セッション内で全ペルソナを演じ分ける** 方式。TeamCreate は使わない（会議のテンポとコスト優先）
- 各発言は必ず「**名前（役割）:**」のプレフィックスで区別する
- 作成者役としての発言は、セッションの親ルールで定められた話し方に従う（本スキル内では指定しない）
- ペルソナの発言と作成者役の発言が混線したら、人間から「整理して」と言われ次第、直近の発言を箇条書きで再整理する

## 終了基準

以下のすべてが満たされたら会議を終了する:
- 全セクションのレビューが完了している
- 人間レビュアーが「終わり」「OK」「解散」と宣言した
- 議事録ファイルが保存され、パスを人間に伝えた
