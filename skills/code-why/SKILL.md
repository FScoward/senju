# code-why — 説明責任ある実装

> 「書いたコードを一行一行説明できるか」  
> これは、単なるルールではなく、本当に理解しているかどうかの試験だ。

コードをコピーして動いた。ドキュメントを見てそれっぽく書いた。LLMが出力したから貼った。  
でも「なぜこの行が必要か」を説明できないなら、そのコードはあなたのものではない。

このスキルは、コードを書くたびに判断の根拠を会話として残す。
コメントとして埋め込むのではなく、対話の中で「なぜ」を明示する。

---

## いつ発動するか / 説明の深さは判断の重さに比例する

コードを書く場面で適用する。ただし、説明の ceremony は変更の重さに合わせる：

| 変更の種類 | 説明の深さ |
|-----------|-----------|
| rename・タイポ修正・import追加 | 「自明なので説明省略」の1行宣言で十分 |
| 数行〜1関数の実装・修正 | 鍵となる判断（1〜3点）のみ |
| アルゴリズム・設計選択を伴う実装 | 方針宣言 + 代替案 + 判断の説明（フルセット） |

trivial な変更で毎回フルの ceremony を出すと、説明がノイズになり、本当に重要な判断が埋もれる。

---

## 説明の構造

コードを書く前後に、以下の構成で会話として説明する。

### 1. 実装前：「方針宣言」

コードを書き始める前に、アプローチの選択理由を述べる。

```
このアプローチを選んだ理由：[WHY]
検討した代替案：
- [案A]：[なぜ退けたか]
- [案B]：[なぜ退けたか]
```

### 2. 実装後：「鍵となる判断の説明」

非自明な行・選択に対して、WHY を会話として添える。

```
L[行番号]：[なぜこの書き方か / 他の書き方ではなくこれを選んだ理由]
```

---

## 何を説明するか

### 必ず説明すべきもの（WHYが重要な判断）

| 判断の種類 | 説明すべき問い |
|-----------|--------------|
| アルゴリズムの選択 | なぜ二分探索か。線形探索ではなく？ |
| データ構造の選択 | なぜ HashMap か。List ではなく？ |
| エラーハンドリング | なぜここで例外を投げるか。握りつぶさない理由は？ |
| 境界値・エッジケース | なぜ `<` ではなく `<=` か |
| 非同期・並行処理 | なぜここで await するか。なぜロックが必要か |
| パフォーマンスの考慮 | なぜキャッシュするか。なぜ遅延評価か |
| 設計パターンの採用 | なぜここで Factory を使うか |
| 可変/不変の選択 | なぜ var か。なぜ val（const）ではないのか |
| null / Option の扱い | なぜここで強制アンラップするか、しないか |
| spec / AC との関係 | この行は AC のどの条項を満たすか、あるいはどこから逸脱・補完しているか |
| 妥協・先送り | なぜ理想解を採らず TODO 化したか、どんな条件で解消するか |

### 説明しなくてよいもの（コード自体が語るもの）

- 単純な変数への代入（`count = 0`）
- 自明なループ変数（`for i in list`）
- 明らかな条件（`if (user == null) return`）
- 命名で意図が明白なもの

**原則**: 「なぜこう書いたか」を他のエンジニアに聞かれたとき、即座に答えられない行は説明する。

## 出力フォーマット

コメントはコードに書かない（コードを汚さない）。  
説明はすべて会話テキストとして、コードブロックの前後に添える。

```
[方針]
〇〇は△△で実装する。
検討した代替案：
- ××：〜の理由で退けた
- □□：パフォーマンス上の懸念で除外

---コードブロック---

[鍵となる判断]
- L3: `Map` を使ったのは O(1) ルックアップが必要なため。
      配列で線形探索すると N 件超えで顕著なボトルネックになる。
- L8: エラーを上流に伝播させている。
      ここで握りつぶすとスタックトレースが消え、デバッグ時に原因が追えなくなるため。
- L12: `copy()` でイミュータブルな更新をしている。
       同一インスタンスを変更すると呼び出し元の状態が予期せず変わるリスクがある。
```

## 永続化レベル

WHY を「会話だけで流す」のか「ファイルに残す」のかは、判断の重さで決める。

| レベル | 該当する判断 | 永続化先 |
|--------|-------------|----------|
| trivial | rename / typo / import 追加 / 既存パターンの単純踏襲 | 会話のみ。記録不要 |
| 中 | 局所判断（アルゴリズム選択、データ構造、エラーハンドリング方針など） | 会話 + **次回コミット時**に `Why: ...` 1 行を末尾に追記（後述「中レベルの追記タイミング」） |
| 大 | **spec / AC からの逸脱 / 妥協 / 先送り（TODO・FIXME 化）/ AC 外のスコープ追加** | **DR ファイル化（下記「保存先分岐」に従う）** |

「大」レベルは会話だけで流すと記録が消える。必ずファイルに残す。

### 中レベルの追記タイミング

T3 実装中は通常まだコミットしていないため、「中」レベルの WHY は以下の順で永続化する。

1. **発生時**: 会話に `[Why-候補] L{行番号}: {1 行の WHY}` の形で書き残す（記録は会話のみ）
2. **コミット直前**: 会話を遡って `[Why-候補]` を収集し、コミットメッセージ末尾の `Why:` セクションに 1 行ずつまとめる

例:

```
feat(alert): add distinct filter to alertReason

Why:
- L42: HashMap 採用（O(1) ルックアップで N^2 を回避するため）
- L51: 例外を上流に伝播（握りつぶすとスタックトレースが消える）
```

これで T3 中に会話で流れた WHY も、コミットで永続化される。コミットを跨ぐ場合は次のコミットに繰り越す。kouunryuusui T5（Push 確認）の時点で `[Why-候補]` が会話に残っていれば、その時点のコミットに amend するか、別途 fixup コミットを作る。

## 保存先分岐（大レベル時）

1. **cwd 配下に `.claude/tmp/decisions/` が存在する場合**
   → kouunryuusui 環境下とみなし、`DR-NNN-{slug}.md` として書き出す
   → kouunryuusui の DR フォーマット（種別 / Spec参照 / 可逆性 / フォローアップ）に従う
2. **存在しない場合（code-why 単体使用）**
   → `.claude/tmp/code-why-log.md` に `WHY-NNN` として追記
   → 同一フォーマット、番号体系のみ独立

いずれの場合も、会話側には `→ DR-NNN 参照` または `→ WHY-NNN 参照` のリンクのみ残し、本文は二重記録しない。

---

## 責任境界（kouunryuusui との連携）

- **code-why**: 行 / 関数レベルの WHY（局所判断）
- **kouunryuusui DR**: ステージレベル（設計 / 分割 / AC 解釈 / PR スコープ）の判断
- 「大」レベルに昇格した code-why の判断は kouunryuusui の DR に一本化する。code-why の会話側には DR への参照リンクのみ残す。

## 説明の質を保つために

**WHAT ではなく WHY を語る**。

> ❌ 「Map を使って重複を除去しています」（WHATの説明）  
> ✅ 「Map を使っているのは、O(1) ルックアップで N^2 のコストを避けるため。データが数千件を超えると Set 実装との差が出てくる」（WHYの説明）

**代替案を明示する**。

ある実装を選ぶことは、他の実装を退けることでもある。  
退けた理由を言語化することで、選択の根拠が初めて明確になる。

**「説明できない行を書かない」という姿勢を保つ**。

書こうとしたが説明できない行があるなら、それは理解が足りていないサイン。  
そのときは立ち止まり、まず理解してから書く。

## このスキルが生まれた背景

あるエンジニアがジュニアのころに学んだこと：

> 「自分が書いたコードを、一行一行説明できることが重要だ」

コピペで動いたコード、雰囲気で書いたコード、誰かの出力をそのまま貼ったコード。  
それらは「動く」かもしれないが、「理解している」とは言えない。

本当に理解したコードだけが、後でデバッグできる。レビューを通過する。  
後任のエンジニアに引き継げる。障害時に即座に原因が分かる。

このスキルは、その姿勢をコードを書くたびに体現するためにある。
