# code-why — 説明責任ある実装

> 「書いたコードを一行一行説明できるか」  
> これは、単なるルールではなく、本当に理解しているかどうかの試験だ。

コードをコピーして動いた。ドキュメントを見てそれっぽく書いた。LLMが出力したから貼った。  
でも「なぜこの行が必要か」を説明できないなら、そのコードはあなたのものではない。

このスキルは、コードを書くたびに判断の根拠を会話として残す。
コメントとして埋め込むのではなく、対話の中で「なぜ」を明示する。

---

## いつ発動するか / 説明の深さは判断の重さに比例する

コードを書く場面で適用する。ただし、説明の ceremony は変更の重さに合わせる：

| 変更の種類 | 説明の深さ |
|-----------|-----------|
| rename・タイポ修正・import追加 | 「自明なので説明省略」の1行宣言で十分 |
| 数行〜1関数の実装・修正 | 鍵となる判断（1〜3点）のみ |
| アルゴリズム・設計選択を伴う実装 | 方針宣言 + 代替案 + 判断の説明（フルセット） |

trivial な変更で毎回フルの ceremony を出すと、説明がノイズになり、本当に重要な判断が埋もれる。

---

## 説明の構造

コードを書く前後に、以下の構成で会話として説明する。

### 1. 実装前：「方針宣言」

コードを書き始める前に、アプローチの選択理由を述べる。

```
このアプローチを選んだ理由：[WHY]
検討した代替案：
- [案A]：[なぜ退けたか]
- [案B]：[なぜ退けたか]
```

### 2. 実装後：「鍵となる判断の説明」

非自明な行・選択に対して、WHY を会話として添える。

```
L[行番号]：[なぜこの書き方か / 他の書き方ではなくこれを選んだ理由]
```

---

## 何を説明するか

### 必ず説明すべきもの（WHYが重要な判断）

| 判断の種類 | 説明すべき問い |
|-----------|--------------|
| アルゴリズムの選択 | なぜ二分探索か。線形探索ではなく？ |
| データ構造の選択 | なぜ HashMap か。List ではなく？ |
| エラーハンドリング | なぜここで例外を投げるか。握りつぶさない理由は？ |
| 境界値・エッジケース | なぜ `<` ではなく `<=` か |
| 非同期・並行処理 | なぜここで await するか。なぜロックが必要か |
| パフォーマンスの考慮 | なぜキャッシュするか。なぜ遅延評価か |
| 設計パターンの採用 | なぜここで Factory を使うか |
| 可変/不変の選択 | なぜ var か。なぜ val（const）ではないのか |
| null / Option の扱い | なぜここで強制アンラップするか、しないか |

### 説明しなくてよいもの（コード自体が語るもの）

- 単純な変数への代入（`count = 0`）
- 自明なループ変数（`for i in list`）
- 明らかな条件（`if (user == null) return`）
- 命名で意図が明白なもの

**原則**: 「なぜこう書いたか」を他のエンジニアに聞かれたとき、即座に答えられない行は説明する。

---

## 出力フォーマット

コメントはコードに書かない（コードを汚さない）。  
説明はすべて会話テキストとして、コードブロックの前後に添える。

```
[方針]
〇〇は△△で実装する。
検討した代替案：
- ××：〜の理由で退けた
- □□：パフォーマンス上の懸念で除外

---コードブロック---

[鍵となる判断]
- L3: `Map` を使ったのは O(1) ルックアップが必要なため。
      配列で線形探索すると N 件超えで顕著なボトルネックになる。
- L8: エラーを上流に伝播させている。
      ここで握りつぶすとスタックトレースが消え、デバッグ時に原因が追えなくなるため。
- L12: `copy()` でイミュータブルな更新をしている。
       同一インスタンスを変更すると呼び出し元の状態が予期せず変わるリスクがある。
```

---

## 説明の質を保つために

**WHAT ではなく WHY を語る**。

> ❌ 「Map を使って重複を除去しています」（WHATの説明）  
> ✅ 「Map を使っているのは、O(1) ルックアップで N^2 のコストを避けるため。データが数千件を超えると Set 実装との差が出てくる」（WHYの説明）

**代替案を明示する**。

ある実装を選ぶことは、他の実装を退けることでもある。  
退けた理由を言語化することで、選択の根拠が初めて明確になる。

**「説明できない行を書かない」という姿勢を保つ**。

書こうとしたが説明できない行があるなら、それは理解が足りていないサイン。  
そのときは立ち止まり、まず理解してから書く。

---

## このスキルが生まれた背景

あるエンジニアがジュニアのころに学んだこと：

> 「自分が書いたコードを、一行一行説明できることが重要だ」

コピペで動いたコード、雰囲気で書いたコード、誰かの出力をそのまま貼ったコード。  
それらは「動く」かもしれないが、「理解している」とは言えない。

本当に理解したコードだけが、後でデバッグできる。レビューを通過する。  
後任のエンジニアに引き継げる。障害時に即座に原因が分かる。

このスキルは、その姿勢をコードを書くたびに体現するためにある。
