# acceptance-test-generator

提供価値・仕様・実装から **ブラックボックス受け入れテストケース** を体系的に生成する。

> **核心原則: テストは「外から観察できる振る舞い」のみを記述する。**
> 内部変数・プライベートメソッド・実装の選択（if 文の数など）に一切依存しない。
> テストが通るかどうかは、実装を完全に差し替えても変わらない。

---

## このスキルと test-matrix の使い分け

| | このスキル | test-matrix |
|---|---|---|
| 出発点 | **ビジネス価値**「誰の何の体験を変えるか」 | 仕様・コード・関数シグネチャ |
| 視点 | ブラックボックス（外側からのみ観察） | ホワイト/グレーボックス可（実装の分岐も見る） |
| テスト種別 | **受け入れテスト**（機能全体が価値を届けるか） | 単体〜統合（入力→出力の技法的網羅） |
| 使い所 | リリース可否判断・QA・ユーザーストーリー検証 | 実装前のテスト設計・カバレッジ向上 |

## ステップ

### Step 1: インプットを収集・整理する

以下を受け取る（複数あれば全部使う。なければ推論する）:

| インプット | 必須 | 説明 |
|---|---|---|
| 提供価値 | 推奨 | 「誰の、何の体験が変わるか」「なぜこの機能が存在するか」 |
| 仕様書・ドキュメント | 推奨 | 要件定義・API仕様・画面仕様・PRD など |
| 実装コード | 推奨 | ソースコード・API定義・DB スキーマ・OpenAPI 等 |

**どれかひとつしかなくても動く**:
- 提供価値のみ → ユーザーストーリーから振る舞いを推論してケースを作る
- 仕様のみ → 仕様に書かれた約束をテストに変換する
- 実装のみ → 外部インターフェースを逆読みしてブラックボックス契約を抽出し、テストを作る。**「実装がこう動いているからこうテストする」ではなく「外から見るとこう見えるはず」でテストを書く**

収集後、以下を抽出して **ユーザーに一度確認する**:

```
対象機能: [機能名 / エンドポイント / 画面 / フロー]
提供価値: [誰の何の体験が変わるか]
外部インターフェース:
  - 入力: [パラメータ・リクエスト・操作]
  - 出力: [レスポンス・画面変化・副作用]
ビジネスルール: [バリデーション・権限・ワークフローの制約]
前提条件: [認証・初期データ・環境]
```

### Step 2: テスト観点を展開する

以下の 8 観点を必ず全部検討する。「該当しない」と判断した場合は理由を明記する。

| # | 観点 | 何を確認するか |
|---|---|---|
| 1 | **正常系（価値実現）** | ユーザーが「うまくいった」と感じる主要シナリオ |
| 2 | **代替正常系** | 正常だが非典型的な入力・フロー（任意項目省略、別フォーマット等） |
| 3 | **入力エラー** | 不正・欠落・型違いの入力に対して意味あるエラーが返るか |
| 4 | **境界値** | min/max・空・null・0・最大長・空配列などの端点 |
| 5 | **権限・認証** | 未認証、権限不足、他ユーザーのリソースへの不正アクセス |
| 6 | **状態依存** | 状態ごとに操作の可否・結果が変わるか（ステータス遷移・順序依存） |
| 7 | **副作用・冪等性** | 同じ操作を 2 回行ったとき何が起きるか。メール/通知の二重送信はないか |
| 8 | **エラーリカバリ** | 失敗後に再試行できるか。データが中途半端な状態にならないか |

### Step 3: テストケースを生成する

各テストケースを以下の形式で出力する:

```
| TC-ID | 観点 | シナリオ名 | 前提条件 | 操作/入力 | 期待結果 | 優先度 |
```

**TC-ID の命名**: `AT-{機能略称}-{連番3桁}` (例: `AT-USR-001`)

**期待結果の書き方のルール**:
- ✅ 「200 OK, body に user_id が含まれる」
- ✅ 「画面に "登録完了" と表示され、DBにレコードが1件増える」
- ❌ 「正しく登録される」（曖昧すぎる）
- ❌ 「UserService.save() が呼ばれる」（内部実装依存）

**優先度の定義**:
| 優先度 | 意味 |
|---|---|
| P1 | ビジネス上クリティカル。失敗 = リリース不可 |
| P2 | 主要ユースケース。通常はリリース前に通す |
| P3 | エッジケース・非典型。余裕があれば通す |

### Step 4: カバレッジ自己チェック

生成後、以下を確認してから出力する:

```
カバレッジチェックリスト:
[ ] 全エンドポイント/機能の正常系（価値実現シナリオ）が1件以上ある
[ ] 全必須入力の欠落ケースがある
[ ] 全バリデーションルールに対応する NG ケースがある
[ ] 認証・権限チェックがある機能は「権限なし」「他ユーザーのリソース」ケースがある
[ ] 状態を持つ機能はすべての状態×イベント組み合わせが網羅されている
[ ] 副作用がある操作（データ変更・通知・課金など）に冪等性テストがある
[ ] P1 ケースのみでビジネス価値が証明できる（P1 だけを実行しても意味ある判断ができる）
```

### Step 5: 出力形式を選んで出力する

ユーザーが指定しなければ **Markdown テーブル（デフォルト）** で出力する。

| オプション | 形式 |
|---|---|
| `markdown` | テーブル形式（デフォルト） |
| `gherkin` | Given/When/Then（BDD/Cucumber 向け） |
| `code:pytest` | pytest コードスケルトン |
| `code:jest` | Jest コードスケルトン |
| `code:rspec` | RSpec コードスケルトン |
| `json` | 構造化 JSON（CI 連携・チケット起票向け） |

詳細な出力形式サンプルは `references/output-formats.md` を参照。

## Gotchas（ハマりやすい罠）

### ブラックボックス原則を守る
実装コードを読むのは「外から見た契約は何か」を把握するためだけ。
「if 文が 3 つある → テストケースが 3 つ必要」という発想はホワイトボックス思考でNG。
「この入力を渡したとき、外から観察できる振る舞いは何か」という視点で一貫する。

### 実装バグを仕様と誤認しない
実装を読んで仕様を逆引きするとき、「これは本当に意図した振る舞いか？」を疑う。
バグを期待値に書いてしまうと、テストがバグを永続化させる。
不明な点はコメントとして「**要確認**: この振る舞いは仕様通りか？」と明記する。

### 組み合わせ爆発を避ける
8 観点 × 全パラメータ × 全状態 を愚直に掛け合わせると爆発する。
同値分割（同じ振る舞いになる入力グループから代表値1つ）で削減する。
100 ケース作っても回されなければ意味がない。P1 で最小・最重要セットを構成する。

### 副作用テストの独立性
DELETE/POST などの副作用を持つテストケースは、実行順序に依存しないよう前提条件を明記する。
「前のテストでデータが作られている前提」は書かない。各テストケースが独立して実行できること。

### 認証トークン・テストデータ準備を前提条件に書く
「ログイン済み」「管理者権限あり」「対象レコードが存在する」などは前提条件に必ず明示する。
曖昧な前提条件があるとテストが再現できなくなる。

### 「期待結果」はユーザーが観察できることだけを書く
HTTP ステータスコード・レスポンスボディ・画面表示・DB の状態変化などは書いてよい。
内部キャッシュの状態・メモリ上の変数・ログ出力（ユーザーに見えないもの）は受け入れテストの期待結果に書かない。

## 出力例

### Markdown テーブル（ユーザー登録 API の例）

```
## 受け入れテストケース: ユーザー登録 POST /api/users

| TC-ID | 観点 | シナリオ | 前提条件 | 入力 | 期待結果 | 優先度 |
|---|---|---|---|---|---|---|
| AT-USR-001 | 正常系 | 有効な情報で登録成功 | DBにメールが存在しない | name="田中", email="valid@example.com", password="Secure123!" | 201 Created, body に user_id が返る | P1 |
| AT-USR-002 | 入力エラー | メールアドレス重複 | DB に "dup@example.com" が存在する | email="dup@example.com" | 409 Conflict, error:"email_already_exists" | P1 |
| AT-USR-003 | 入力エラー | 必須項目(name)欠落 | - | name 省略, email, password あり | 400 Bad Request, error:"name_required" | P1 |
| AT-USR-004 | 境界値 | パスワード最小長(8文字)ちょうど | - | password="Abcd123!" (8文字) | 201 Created | P2 |
| AT-USR-005 | 境界値 | パスワード7文字(最小長未満) | - | password="Abcd12!" (7文字) | 400 Bad Request, error:"password_too_short" | P1 |
| AT-USR-006 | 権限 | 認証なしでアクセス | - | Authorization ヘッダーなし | 401 Unauthorized | P1 |
| AT-USR-007 | 副作用 | 同じリクエストを2回送信 | - | 同一メールで2回 POST | 1回目: 201, 2回目: 409. DBのレコードは1件のみ | P2 |
```

### Gherkin（BDD 形式）

```gherkin
Feature: ユーザー登録

  Background:
    Given API エンドポイント "POST /api/users" が利用可能

  Scenario: AT-USR-001 有効な情報で登録成功
    Given DB に "valid@example.com" が存在しない
    When POST /api/users に {"name":"田中","email":"valid@example.com","password":"Secure123!"} を送る
    Then レスポンスは 201 Created
    And レスポンスボディに "user_id" フィールドが含まれる

  Scenario: AT-USR-002 メールアドレス重複
    Given DB に "dup@example.com" が存在する
    When POST /api/users に {"email":"dup@example.com"} を送る
    Then レスポンスは 409 Conflict
    And error は "email_already_exists"
```
