# init-prompt $ARGUMENTS

タスクを実行する前に、ゴール・制約・完了基準の3点セットでプロンプトを構造化する。

## なぜ3点セットが必要か

曖昧なプロンプトで実装を始めると：

- **ゴールのズレ**: 「思っていたものと違う」が実装後に発覚する
- **制約の見落とし**: 変えてはいけないAPIや既存コードを壊す
- **完了基準の不在**: どこまでやればいいかが曖昧なまま進む

3点セットを最初に揃えることで、認識のズレを事前に解消できる。

---

## 出力テンプレート

```
Goal: <達成したいこと>
Constraints: <守るべき制約>
Acceptance criteria:
  - <判定基準1>
  - <判定基準2>
  - ...
```

---

## 進め方

### Step 1: 現状の把握

ユーザーのメッセージ・会話の文脈から、既に分かっている情報を整理する：

| フィールド | 確認するポイント | 埋まっていない場合 |
|-----------|----------------|-----------------|
| **Goal** | 「〜したい」「〜を作りたい」などの動詞句 | Step 2 へ |
| **Constraints** | 「〜を変えずに」「〜のAPIを使って」「技術スタックや期限」など | 「特になし」と仮置き |
| **Acceptance criteria** | 「〜できれば完了」「〜のテストが通れば」など | Step 2 へ |

### Step 2: ヒアリング（情報が足りない場合）

1回の質問は **最大2〜3個まで**。全部を一度に聞かない。最も重要な欠けているピースから聞く。

**Goal が不明な場合**
- 「このタスクで一番達成したいことは何ですか？」
- 「完成したとき、ユーザーは何ができるようになりますか？」

**Constraints が不明な場合**
- 「変更してはいけないもの、または必ず使わなければならない技術はありますか？」
- 「パフォーマンス要件や後方互換性の制約はありますか？」
- （答えが「特になし」でも問題ない。この欄は省略せず「特になし」と記載する）

**Acceptance criteria が不明な場合**
- 「タスクが完了したと判断できる条件を教えてください」
- 「どんな動作やテストが通れば "完了" と言えますか？」

### Step 3: 構造化プロンプトの出力

3点セットが揃ったら、以下の形式で出力する：

```
Goal: <達成したいこと>
Constraints: <守るべき制約（なければ「特になし」）>
Acceptance criteria:
  - <判定基準1>
  - <判定基準2>
  - ...
```

出力後に「この内容でよいですか？」と確認する。
OKなら、次のアクション（実装・スキル起動など）を提案する。

---

## ヒアリングのルール

1. **ユーザーの言葉をそのまま使う** — 勝手に言い換えると意図が変わる
2. **仮説を立てて確認する** — 「〜ということでしょうか？」で時短できる
3. **Constraints は「特になし」でも進む** — 空白のまま放置しない
4. **完璧を求めない** — 今わかっていることを整理すれば十分。要求は変わる

---

## 他スキルとの連携

このスキルで整理した3点セットは、そのまま以下のスキルのインプットとして使える：

| スキル | 渡す情報 |
|--------|---------|
| `kouunryuusui` | Goal → Epic説明、AC → チケット検証基準 |
| `vertical-slice` | Goal → Walking Skeleton の設定 |
| `software-requirements` | 3点セット全体 → SRS起点（より深い要件整理が必要な場合） |
| `mihari` | AC → テスト観点の起点 |
| `sier-dev` | Goal + Constraints → 設計書の起点 |

## 出力例

### 例1: 機能追加タスク

```
Goal: 従業員一覧画面にCSVエクスポート機能を追加する
Constraints:
  - 既存のDBスキーマを変更しない
  - バックエンドはKotlin + Spring Boot
  - エクスポートは管理者ロールのみ許可
Acceptance criteria:
  - 管理者が「エクスポート」ボタンを押すとCSVがダウンロードできる
  - CSVには氏名・メール・部署・役職が含まれる
  - 一般ユーザーにはボタンが表示されない
  - エクスポート数1000件で5秒以内に完了する
```

### 例2: バグ修正タスク

```
Goal: 検索APIでNullPointerExceptionが発生する不具合を修正する
Constraints:
  - 公開APIのレスポンス形式を変えない
  - 既存テストを壊さない
Acceptance criteria:
  - 検索キーワードが空文字の場合に500エラーが返らない
  - 既存のテストがすべて通る
  - 修正後のユニットテストが追加されている
```
