# SIer開発スキル $ARGUMENTS

SIerスタイルのウォーターフォール型開発プロセスをサポートする。各フェーズで必要なドキュメントをMarkdownで生成し、`docs/` フォルダに保存する。

## ドキュメント体系

```
docs/
├── 01_requirements/
│   └── 要件定義書.md
├── 02_basic-design/
│   └── 基本設計書.md
├── 03_external-design/
│   └── 外部設計書.md
├── 04_internal-design/
│   └── 内部設計書.md
└── 05_test/
    ├── 全体テスト計画書.md
    ├── 単体テスト仕様書.md
    └── 結合テスト仕様書.md
```

## フェーズと対応ドキュメント

| フェーズ | ドキュメント | 依存 | テンプレート |
|---------|------------|------|------------|
| 要件定義 | 要件定義書 | なし | `references/requirements.md` |
| 基本設計 | 基本設計書 | 要件定義書 | `references/basic-design.md` |
| 外部設計 | 外部設計書 | 基本設計書 | `references/external-design.md` |
| 内部設計 | 内部設計書 | 外部設計書 | `references/internal-design.md` |
| テスト計画 | 全体テスト計画書 | 基本設計書 | `references/overall-test-plan.md` |
| 単体テスト | 単体テスト仕様書 | 内部設計書 | `references/unit-test.md` |
| 結合テスト | 結合テスト仕様書 | 外部・内部設計書 | `references/integration-test.md` |

## 作業フロー

### Step 1: フェーズを特定する

ユーザーのリクエストからどのフェーズ・ドキュメントを作るか判断する：

- **「新しいシステムを作りたい」「ゼロから始める」** → 要件定義フェーズから開始
- **特定のドキュメント名が明示されている** → そのドキュメントを作成
- **フェーズが不明確** → 以下の選択肢を提示して確認する：
  1. 要件定義書（ゼロから）
  2. 基本設計書（要件は決まっている）
  3. 外部設計書 / 内部設計書（基本設計は済んでいる）
  4. テストドキュメント（設計は済んでいる）

### Step 2: 依存ドキュメントを確認する

作成するドキュメントが依存する上位ドキュメントが `docs/` に存在するか確認する：

- **存在する** → 内容を読み込み、一貫性・トレーサビリティを保って生成する
- **存在しない** → ユーザーに確認する：
  - 先に上位ドキュメントを作るか
  - 情報をヒアリングしてそのまま作るか

### Step 3: 情報をヒアリングする

テンプレートを参照しながら、不足している情報を最小限の質問で収集する。一度に聞きすぎない（2〜3個ずつ）。

テンプレートは `references/` フォルダの各ファイルを読み込む（上の表を参照）。

### Step 4: ドキュメントを生成・保存する

テンプレートとヒアリング内容を元にドキュメントを生成し、`docs/` フォルダに保存する。

**生成ルール：**
- 空のセクションは `TBD（未定）` と記載し、決める必要がある内容をコメントで補足する
- 依存ドキュメントが存在する場合、内容と矛盾しないよう一貫性を保つ
- 図・ダイアグラムはMermaidで表現する
- テスト仕様のテストケースにはIDを付与する（例: TC-001）

### Step 5: 次フェーズを提案する

ドキュメント生成後、次に作成すべきドキュメントを提案する：

```
✅ 基本設計書 を docs/02_basic-design/基本設計書.md に保存しました。

次のステップ：
- 外部設計書（画面・インターフェース設計）
- 内部設計書（モジュール・DB設計）
- 全体テスト計画書

どれから進めますか？
```

## 各ドキュメントのポイント

### 要件定義書
- 「なぜ作るか（Why）」を最重要視する
- 機能要件と非機能要件を明確に分ける
- 制約条件・前提条件を明記する

### 基本設計書
- 要件定義書の機能要件との対応（トレーサビリティ）を明記する
- システム構成図はMermaidで描く
- 機能一覧・画面一覧・データフローを含める

### 外部設計書
- 画面レイアウトはASCIIアートまたはMermaidで表現する
- 各画面の入出力項目・バリデーションを明記する
- 帳票・インターフェース仕様も含める

### 内部設計書
- クラス図・シーケンス図はMermaidで描く
- 処理フローは番号付きリストで記述する
- DB設計はテーブル定義（カラム名・型・制約）を表形式で記載する

### テストドキュメント
- テストケースにはIDを付与する（TC-001, IT-001 など）
- 合否判定基準を具体的に記載する
- テストデータは具体的な値を記入する
