# TDDリファクタリング要件 & テスト品質基準

> このファイルは下位フロー T3 から参照される。パスA（Team並列）・パスB（単一エージェント）共通。
> 実装エージェントのプロンプトにこの内容を含めること。

## Iron Law（絶対ルール）

**テスト前にプロダクションコードを書いた場合: 削除して最初からやり直せ。**

- 「参考に残す」は不可。削除 = 削除
- 「既に X 時間かけた」はサンクコスト。残すほうが技術的負債
- 「後でテストを書けば同じ」→ 誤り。テスト前は「何をすべきか」の確認。テスト後は「何をしたか」の確認。目的が異なる
- 「今回は特別」→ Anti-rationalization。`references/quality-rules.md` の Anti-rationalization パターン集を参照

この Iron Law に疑問を感じたら → `references/quality-rules.md` を読め

## TDDリファクタリング要件（省略禁止）

TDDの Red → Green → **Refactor** サイクルにおいて、**Refactorステップを飛ばしてはならない**。
「テストが通ったからコミット」ではなく、「テストが通った → リファクタリングした → コミット」が正しい流れ。

### Red（テスト作成）
- ACの各項目に対応するテストを書く
- **正常系だけでなく、異常系・境界値のテストも最初から書く**
- テストが失敗することを確認してから次のステップに進む

### Green（最小実装）
- テストを通す最小限のコードを書く
- この段階では「動くこと」が最優先。きれいさは次のステップ

### Refactor（品質改善・省略禁止）

以下の観点で必ずリファクタリングを行う:

| 観点 | チェック内容 | 例 |
|------|------------|-----|
| **重複排除** | 同じロジックが2箇所以上にないか | 共通メソッドへの抽出 |
| **命名改善** | 変数名・関数名が意図を正確に伝えるか | `data` → `surveyResponses` |
| **責務分離** | 1つのクラス/関数が複数の責務を持っていないか | UseCase内のバリデーションをDomain層に移動 |
| **マジックナンバー排除** | ハードコードされた値がないか | `0.70` → `MINIMUM_ANSWER_RATE` |
| **不要な複雑さ** | 過度なネスト、長いメソッド、不要な抽象化がないか | early returnで条件分岐を簡素化 |
| **既存パターンとの整合** | コードベースの既存実装パターンに合っているか | 既存のRepository実装パターンに従う |

## コミット粒度ガイドライン

| フェーズ | コミットメッセージ形式 | 含めるもの |
|---------|---------------------|-----------|
| Red | `test: {ACに対応するテスト内容}` | テストコードのみ |
| Green | `feat: {実装内容}` or `fix: {修正内容}` | 実装コード + テストの微修正 |
| Refactor | `refactor: {リファクタリング内容}` | リファクタリング変更のみ（テストは変えない） |

**重要**: Green → Refactor の間にコミットしない。Refactorまで完了してからコミットする。
ただしRefactorの変更が大きい場合はGreenとRefactorを分けてコミットしてよい。

## テスト品質の要件

- 正常系テスト: ACの全項目をカバー（100%必須）
- 異常系テスト: 想定されるエラーケースをカバー
- 境界値テスト: 数値の上下限、空文字列、null、最大長等
- テストには**意味のあるassertion**が必須（`assertTrue(true)` のような空テストは禁止）

### ユニットテスト必要十分チェック（QGで確認）

ユニットテストは「ACをカバーしているか」だけでなく、「適切な粒度で書かれているか」を確認する。

| チェック項目 | OK例 | NG例 |
|------------|------|------|
| **外部依存はモックされているか** | `mock(UserRepository::class)` でDBを差し替え | 本物のDBを呼ぶテストをunitとして分類 |
| **1テスト = 1振る舞い** | `test("空メールはバリデーションエラー")` | 1テストで正常系・異常系を全部確認 |
| **ドメインロジックに直接触れているか** | UseCase / Domain層を直接テスト | Controllerテストだけでドメインロジックを間接的にカバー |
| **assertionが具体的か** | `assertEquals(ErrorCode.EMAIL_REQUIRED, result.error)` | `assertNotNull(result.error)` だけ |
| **統合テストとの役割分担** | unitでロジック検証 + integrationでAPI結合検証 | 統合テストのみでunitを代替 |

#### テストピラミッド確認

sprint-contract.mdの「テストスコープ」に従い、各層のテストが揃っているかを確認:

| 層 | 目的 | 最低基準 |
|----|------|---------|
| Unit | ドメインロジック・バリデーション・変換処理の単体検証 | ACの正常系・異常系・境界値をカバー |
| Integration | レイヤー間・外部サービスとの結合検証 | 主要APIエンドポイント・DBアクセスをカバー |
| E2E | ユーザーシナリオの端対端検証 | spec.mdにE2Eスコープが明記された項目のみ |

> **⚠️ 統合テストはユニットテストの代替にならない**
> 統合テストが通っていてもユニットテストがなければ「どのロジックが壊れたか」が特定できない。
> ドメインロジック・バリデーション・副作用（イベント発行等）は必ずユニットテストで検証する。
