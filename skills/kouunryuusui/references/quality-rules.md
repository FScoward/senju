# 横断的品質ルール

すべてのフェーズで適用する品質ルール集。フロー実行開始時に読み込み、各フェーズの判断基準として使用する。

---

## Verification Before Completion 鉄則

> 証拠なしに成功を主張するな。

```
BEFORE claiming PASS / 完了 / 成功:

1. IDENTIFY: 何のコマンドがその主張を証明するか？
2. RUN: そのコマンドをこのメッセージで実行する
3. READ: 出力を読み、exit code を確認する
4. ONLY THEN: 主張する

いずれかのステップを飛ばす = 虚偽報告
```

### 適用タイミング（必須）

- QG-PASS を宣言する前
- T3 完了を宣言する前
- T5 Go 判定を行う前
- エージェントが「成功」と報告したとき（エージェントの報告を信頼しない。VCS diff で独立に検証する）

### 検証コマンドの例

| 主張 | 必要な証拠 | 不十分な証拠 |
|------|-----------|------------|
| テスト全PASS | テストコマンドの出力（failures: 0） | 「前回はパスした」「パスするはず」 |
| ビルド成功 | ビルドコマンド exit 0 | lintパス（≠ コンパイル成功） |
| バグ修正完了 | 修正後にテストが通ること | コードを変えた |
| エージェント完了 | VCS diff で変更を確認 | エージェントが「DONE」と言った |
| sprint-contract.md 全基準✅ | 各基準のコマンド実行結果 | 「実装したから満たしているはず」 |

## Anti-rationalization パターン集

以下の考えが浮かんだら、**そのフレーズ自体がアンチパターンのシグナル**である。

| こう思ったら | 現実 |
|------------|------|
| 「小さな変更だからQG簡略化でいい」 | 小さな変更こそ既存コードへの副作用リスクがある |
| 「既にコードがあるからE2不要」 | 修正の影響波及調査のためにE2は必須 |
| 「テストが既にあるからT3の検証基準定義不要」 | 修正後の期待動作を定義するために必須 |
| 「ビルドが通った → 品質OK」 | ビルド通過 ≠ コード品質 |
| 「ACに書いていないエッジケースはテスト不要」 | 自発的にエッジケースを追加するのが品質原則 |
| 「今回は特別な事情がある」 | ない |
| 「エージェントが成功と言った」 | 独立に検証せよ |
| 「後でテストを書けば同じ目標を達成できる」 | テスト前は「何をすべきか」の確認。テスト後は「何をしたか」の確認。目的が異なる |
| 「修正ループが3回で収束したからOK」 | 収束回数ではなく、実際に全チェックがパスしているかが基準 |
| 「sprint-contract.mdの基準が曖昧だから判定が難しい」 | 基準が曖昧ならDecision Recordを書いて明確化してから判定する |
| 「統合テストが通っているからユニットテストは不要」 | 統合テストはユニットテストの代替にならない。ドメインロジック・バリデーション・副作用は必ずユニットテストで検証する |
| 「テストが存在するから充足している」 | テストの存在 ≠ 充足。正常系・異常系・境界値が揃っているか、assertionが具体的かを確認する |
| 「カバレッジが高いからテスト品質は問題ない」 | カバレッジはテストの網羅性を保証しない。意味のないassertionでカバレッジは上がる |
| 「1回レビューして指摘なしだから十分」 | 単一エージェントは観点の盲点がある。`mihari` の3並列×収束ループで複数Round確認するまで PASS と主張しない |
| 「Warning だから次工程に進んでいい」 | Warning も Decision Record なしで通過は禁止。対応 or 明示的な理由記録のいずれかを必須とする |
| 「mihari が N 回ループしたから十分」 | 回数ではなく「指摘が出なくなったか + 追加テストが実行で PASS するか」が基準。Round 数を根拠に PASS を宣言してはならない |
| 「テスト追加エージェントが DONE と言った」 | 独立に VCS diff で確認し、テスト実行コマンドで exit 0 を確認するまで追加完了と認めない |
