# 受け入れ条件（A.C.）書き方ガイドライン

チケットに記載するA.C.の品質基準とフォーマットを定義する。
E1（品質判定）とE4（AC生成）の両方で参照される。

---

## 必須フォーマット: GWT + Examples（SBE形式）

すべてのA.C.は **Given-When-Then + Examples テーブル** で記述する。

```markdown
- [ ] **AC{N}: {条件のタイトル}**
  - **Given** {具体的なデータ値を含む前提条件}
  - **When** {具体的な操作（API呼び出し、UI操作、バッチ実行等）}
  - **Then** {具体的な期待結果（レスポンス値、UI状態変化、DB状態等）}

  **Examples:**
  | ケース | 入力 | 初期状態 | 操作 | 期待結果 |
  |--------|------|---------|------|---------|
  | 正常系 | {具体値} | {具体値} | {操作} | {具体的な結果} |
  | 境界値 | {具体値} | {具体値} | {操作} | {具体的な結果} |
  | 異常系 | {具体値} | {具体値} | {操作} | {エラー内容} |
```

## A.C.の品質基準（7項目チェックリスト）

### 1. 具体性（Specific）

- **NG**: 「正しく表示される」「適切に処理される」「エラーが出る」
- **OK**: 「ステータスコード200で`{ "id": 1, "name": "..." }`が返る」「バリデーションエラー`NAME_REQUIRED`が返る」

> 判定: GWTの各節に**具体的なデータ値・状態・結果**が含まれているか

### 2. テスト可能性（Testable）

- **NG**: 「ユーザーにとって分かりやすい」「パフォーマンスが良い」
- **OK**: 「レスポンスタイム95パーセンタイルが200ms以下」「WCAG 2.1 AA基準を満たす」

> 判定: 第三者が読んで**PASS/FAILを機械的に判定できる**か

### 3. 正常系の網羅性（Happy Path Coverage）

- 主要なユースケースが全てACとして記述されているか
- 1つのACに複数の独立した振る舞いを詰め込んでいないか（1AC = 1振る舞い）

> 判定: ユーザーストーリーの「〜したい」が**全てACに対応**しているか

### 4. 異常系・エラー系（Error Scenarios）

必須の異常系パターン:
- **バリデーションエラー**: 必須項目未入力、型不正、範囲外、最大長超過
- **認証・認可エラー**: 未認証、権限不足、トークン期限切れ
- **リソース不存在**: 存在しないID指定、削除済みリソースへのアクセス
- **競合**: 楽観ロック違反、重複登録

> 判定: 上記パターンのうち**該当するもの全てにACがある**か

### 5. 境界値（Boundary Values）

必須の境界値パターン:
- **数値**: 0、1、最大値、最大値+1、負数
- **文字列**: 空文字、1文字、最大長、最大長+1、マルチバイト文字
- **コレクション**: 空リスト、1件、上限件数、上限+1
- **日時**: 過去日、現在、未来、境界日時（月末、年末、うるう年）

> 判定: 入力パラメータの**境界値がExamplesテーブルに含まれている**か

### 6. 副作用の明示（Side Effects）

APIやUI操作の「直接の結果」だけでなく、**副作用**もACに含める:
- **DB変更**: どのテーブルのどのレコードがどう変わるか
- **イベント発行**: どのイベントがpublishされるか
- **通知**: メール・Slack・プッシュ通知が送られるか
- **監査ログ**: ログに何が記録されるか
- **キャッシュ**: キャッシュのinvalidationが発生するか

> 判定: 操作の結果、**Then以外に起きるべきことが全て記述されている**か

### 7. 非機能要件（Non-Functional Requirements）

該当する場合のみ。全てのACに必須ではない:
- **パフォーマンス**: レスポンスタイム、スループット
- **セキュリティ**: 入力サニタイズ、SQLインジェクション防止、XSS防止
- **冪等性**: 同じリクエストを2回送っても安全か
- **並行性**: 同時実行時の振る舞い

## A.C.過不足分析プロセス

E1で既存のA.C.を分析し、不足を補完するための手順。

### Step 1: フォーマット検査

| チェック | 判定 |
|---------|------|
| GWT形式で書かれているか | 形式不備 → GWT形式に書き直し |
| Examplesテーブルがあるか | なし → 追加 |
| 各GWT節に具体値があるか | 抽象的 → 具体値に置き換え |

### Step 2: 網羅性検査（7項目チェックリスト）

上記7項目を順にチェックし、不足項目を洗い出す:

```markdown
## A.C.品質レポート

| # | チェック項目 | 状態 | 不足内容 |
|---|------------|------|---------|
| 1 | 具体性 | ✅/⚠️/❌ | {不足があれば記述} |
| 2 | テスト可能性 | ✅/⚠️/❌ | {不足があれば記述} |
| 3 | 正常系網羅性 | ✅/⚠️/❌ | {不足があれば記述} |
| 4 | 異常系・エラー系 | ✅/⚠️/❌ | {不足があれば記述} |
| 5 | 境界値 | ✅/⚠️/❌ | {不足があれば記述} |
| 6 | 副作用の明示 | ✅/⚠️/❌ | {不足があれば記述} |
| 7 | 非機能要件 | ✅/⚠️/❌ | {該当なし or 不足内容} |

**総合判定**: PASS / NEEDS_SUPPLEMENT
**補完が必要なAC数**: {N}件
```

### Step 3: 自動補完

不足項目に対して、以下の優先順位で補完する:

1. **仕様書/PRDから推定**: リンクされたドキュメントから具体値を抽出
2. **コードベースから推定**: 既存の実装・テスト・バリデーションルールから制約を抽出
3. **ドメイン知識から推定**: 一般的なビジネスルール・技術制約から推定（Decision Recordに記録）

補完したACには `[補完]` マークを付け、元のACと区別する:

```markdown
- [ ] **AC3: バリデーションエラー [補完]**
  - **Given** ユーザーが登録フォームを開いている
  - **When** 名前を空のまま送信する
  - **Then** バリデーションエラー `NAME_REQUIRED` が表示される

  > 補完理由: 元のACに異常系が含まれていなかったため、
  > UserEntity のバリデーションルール（name: not blank）から推定して追加
```

## アンチパターン

### 避けるべきA.C.の書き方

| アンチパターン | 問題 | 改善例 |
|--------------|------|--------|
| 「正しく動作する」 | テスト不能 | 具体的な入出力を記述 |
| 「エラーハンドリングする」 | 何のエラーか不明 | エラー種別ごとにACを分ける |
| 「UIが更新される」 | 何がどう変わるか不明 | 変更前後の状態を明記 |
| 1ACに5つのThen | 粒度が粗すぎる | 1AC = 1振る舞いに分割 |
| Examplesなし | 境界値が不明 | 正常/境界/異常の具体例を追加 |
| 「〜できる」のみ | できない場合が不明 | できない場合のACも追加 |
