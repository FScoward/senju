# 行雲流水（kouunryuusui） $ARGUMENTS

Epic/USチケットを起点に、**設計→チケット分割→実装→PR作成**まで自律的に走り続ける開発ワークフロー。

## 実行環境互換（Claude / Codex）

このスキルは Claude Code と Codex の両方から呼ばれることを前提にする。本文中に Claude Code 固有の tool 名が出てきても、Codex では以下の adapter で同等の契約を満たす。

| 概念 | Claude Code での実行 | Codex での実行 |
|------|---------------------|----------------|
| `Skill(name)` | Claude の Skill 呼び出し | 該当スキルの `SKILL.md` を読み、そのプロトコルを明示的に実行する |
| `Agent(...)` / `subagent_type` | Claude の Agent / Native Team | 利用可能な sub-agent / multi-agent tool があれば使う。無ければメインセッションで順次実行し、handoff artifact に状態を残す |
| `TeamCreate` / `SendMessage` / `TaskCreate` | Claude Code Native Team | Codex で同名 tool が無い場合は `.claude/tmp/team/` と `.claude/tmp/handoffs/` のファイルでタスク状態・進捗・ブロッカーを管理する |
| `advisor()` | Claude の advisor tool | `codex-advisor` スキルまたは利用可能なセカンドオピニオン手段で代替する。代替不可なら DR に記録し、Tier 2/3 ではユーザーにエスカレーションする |
| `run_in_background: true` | Claude のバックグラウンド Agent 実行 | Codex では並列 tool / sub-agent が使える場合のみ並列化する。使えない場合は独立タスクを順次実行し、順次化した理由を `scratch.md` に残す |

`.claude/tmp` はこのワークフローの互換 artifact namespace として扱う。Claude 専用ディレクトリではない。既存の `qg-required` hook、DR、handoff、final-report がこのパスを参照するため、Codex 実行時も `.claude/tmp` に統一して書き出す。

> **重要**: Codex で tool 名を読み替える場合も、QG の「スキルを呼んだことにする」のは禁止。`mihari` / `review-loop` などは該当スキルの手順を実際に実行し、その実行証跡を `qg-result.md` に記録する。

## 核心原則

> **品質を最優先にせよ。時間をかけてでも正しく作れ。迷ったらDecision Recordを書いて進め。**

1. **品質は速度に優先する**: リファクタリングを省略しない。テストは境界値・異常系まで書く。コードレビュー指摘は全て対応する
2. **自律的に判断し、品質を落とさず進む**: 上位フロー E1〜E3 + 下位フロー T1〜QG は自律実行。選択肢がある場合はDecision Recordを作成し、最善の選択をして進む
3. **品質チェック失敗は必ず修正する**: QG内の自己修正ループで自動修正（最大5回）。ビルド通過だけでなくコード品質も担保する
4. **T5のみ停止**: ユーザーの明示的承認を待つ（pushは人間の判断）。T3完了後に「PR作りますか？」と聞いてはならない。T3→QG→T5は一気通貫
5. **ハーネスの定期見直し**: 各フェーズはモデルが単独では実行できない点についての仮定を符号化している。モデル改善により仮定が陳腐化する可能性を意識し、定期的にフローの各コンポーネントの必要性を見直す
6. **修正タスクでもフローを守る**: 既存コードへの修正・バグ修正・仕様変更でも全ステップを実行する。各ステップには修正時の固有の目的がある（影響調査、整合性チェック、回帰防止等）。「既に存在する」はスキップの理由にならない
7. **ACブラッシュアップは実装の前提条件**: GWT+Examples形式・7項目網羅性を満たしていないACで実装を始めてはならない。上位フロー（E1）を通らない下位フロー直接実行でも **T0（AC品質チェック）を必ず実施する**。形式不備のACをそのまま sprint-contract.md に転記しない
8. **作業根拠を明示する**: 各フェーズ完了時に「今どの情報を根拠に作業しているか」を `scratch.md` のSSOT宣言として記録する（E1完了→`spec-draft.md`、E2完了→`design.md`、E3完了→`ticket-plan.md`）。Jira/Notionとローカルで内容が食い違う場合は現フェーズのSSOTファイルを正とする

## advisor 使用ポリシー

> **advisor は「次の大きな決定を下す直前」に呼ぶ。設計後・実装後ではなく、設計前・分割前・Push前。**

`advisor` はセッション全体のコンテキストを見る上位レビュアー。高リスクな判断ポイントで呼び出し、セカンドオピニオンを得る。Claude では `advisor()` tool、Codex では `codex-advisor` スキルまたは利用可能な同等手段に読み替える。

> **Tier 定義（`references/upper-flow.md` の「タスク性質判定」で決定）:**
> - **Tier 1**: CRUD・設定変更・既知パターン。変更ファイル3未満
> - **Tier 2**: 複数コンポーネント・UI・分岐多い。変更ファイル3〜15
> - **Tier 3**: 既存統合深い・パフォーマンス・ドメイン知識必要。変更ファイル15+ or 不明

- 具体的な懸念点が挙がった場合は、その点を修正してから次フェーズへ

- advisor の指摘とコードベース調査が矛盾する場合は、もう一度 advisor を呼んで reconcile を依頼する

### 品質原則（全フェーズ共通）

- **リファクタリングは省略禁止**: TDDのRefactorステップを飛ばさない。Green後に必ず重複排除・命名改善・責務分離を行う
- **テストは正常系だけでは不十分**: 異常系・境界値・エッジケースのテストも必須。ACに明記されていないエッジケースも自発的に追加する
- **コードレビュー指摘は全て対応**: Critical/Warningとも修正する。Infoは判断に基づき対応
- **「動けばいい」は品質ではない**: 可読性・保守性・テスト容易性を意識し、将来の開発者が理解しやすいコードを書く
- **急いで雑にするより、丁寧に一回で正しく作る**: 手戻りコストは初回の丁寧さより常に高い

## 並列実行の原則

- 独立した作業は、実行環境が提供する並列機構（Claude の `run_in_background: true` / Native Team、Codex の sub-agent / multi-agent tool 等）で並列化する
- BE+FE両方の変更がある場合は、利用可能なら Team / sub-agent で並列実装する。利用不可なら順次実行し、順次化した理由と順序を `scratch.md` に残す
- **E4の複数チケット並列実行**: Claude では Native Team（`TeamCreate` / `SendMessage`）、Codex では利用可能な sub-agent / file-handoff でワーカーを起動し、進捗共有・コンフリクト予防・段階的エスカレーションを行う
- メインのコンテキストウィンドウをクリーンに保つため、各フェーズの実作業はサブエージェントに委譲する

## Decision Record（横断メカニズム）

以下のいずれかが発生したら、ユーザーに聞かず Decision Record を残して進む。**(b)〜(d) は AI 側が能動的にセルフ検知する義務がある。**

- (a) 判断が必要になった（曖昧で進めない、選択肢が複数ある）
- (b) **spec / AC に明記されていない判断をした**（推定・補完を含む）
- (c) **spec から逸脱・妥協・先送り（TODO/FIXME 化）した**
- (d) **AC 外の追加変更・スコープ拡張をした**

`通常判断` と `spec 外`（明示外 / 妥協 / 先送り / スコープ追加）は **種別** フィールドで区別する。後述の trivial 除外基準に該当するものは DR 不要。

### 保存先

worktreeルートの `.claude/tmp/decisions/` ディレクトリに保存する。Codex から実行している場合も同じ場所に保存する:
```
.claude/tmp/decisions/
  DR-001-scope-split.md
  DR-002-ec-strategy.md
  DR-003-ac-ambiguity.md
```

### フォーマット

> **Gotcha**: `### なぜ他の案を採らなかったか` は省略禁止。却下案の理由は「負の空間」の記録であり、
> これを残さないと後で同じ案が再提案されたときに判断を一からやり直すことになる。
> 「デメリット列に書いたから十分」は不可。そこは対比のための一語であり、退けた理由の説明ではない。

```markdown
# DR-{番号}: {タイトル}

- **Phase**: {E1〜E4 or T1〜T5/QG}（{フェーズ名}）
- **日時**: {ISO 8601}
- **種別**: 通常判断 | 明示外 | 妥協 | 先送り | スコープ追加
- **Spec参照**: {AC番号 / 仕様パス:行 / "該当なし"}
- **可逆性**: 高（後で変更容易） | 中 | 低（後戻りコスト大）
- **フォローアップ**: {TODOチケット番号 or "不要"}
- **状況**: {何を判断する必要があったか}

## 選択肢

| # | 選択肢 | メリット | デメリット |
|---|--------|---------|----------|
| 1 | {案A} | {メリット} | {デメリット} |
| 2 | {案B} | {メリット} | {デメリット} |

## 決定

**選択: {番号}** — {選択した案}

## 理由

{なぜこの選択が最適か。コードベースの調査結果、既存パターンとの整合性、リスク分析を含む}

### なぜ他の案を採らなかったか（必須）

却下した各案について、退けた理由を 1 行ずつ残す。

- {案B}: {なぜ退けたか}
- {案C}: {なぜ退けたか}

> 代替案が無かった場合は「代替案なし（{なぜ一意に決まったか}）」と明記する。空欄禁止。
```

### 判断基準（AIが最適解を選ぶ指針）

1. **既存パターンとの整合性**: コードベースの既存実装に合わせる
2. **最小変更原則**: 変更量が少ない方を選ぶ
3. **安全側に倒す**: リスクが不明な場合は保守的な選択
4. **チケットのACに忠実**: 受け入れ条件に明記されていることを優先
5. **可逆性**: 後から変更しやすい選択を優先

### Worked Examples（種別の判断を安定させるための実例）

**例 1: 種別「明示外」**

```markdown
# DR-007: 期間フィルタの上限 90 日制約を追加

- **Phase**: E1（AC 品質チェック）
- **日時**: 2026-05-20T10:42:00+09:00
- **種別**: 明示外
- **Spec参照**: APP-1234 AC2（期間フィルタの動作）— 上限の記載なし
- **可逆性**: 高（API 引数バリデーションを緩めるだけで戻せる）
- **フォローアップ**: 不要（PR description に明記）
- **状況**: AC2 に「期間でフィルタできる」とあるが、上限期間の規定がない。実装上は性能観点で上限が必須。

## 選択肢

| # | 選択肢 | メリット | デメリット |
|---|--------|---------|----------|
| 1 | 上限を 90 日に固定 | 既存の集計バッチと同じ上限で整合 | AC に明記がないので利用者の期待とズレる可能性 |
| 2 | 上限なし（DB タイムアウトに委ねる） | AC 通り | 大量データで OOM / タイムアウトリスク |

## 決定

**選択: 1** — 上限 90 日

## 理由

既存の `ep_aggregation` バッチが 90 日上限で動いており、整合性が取れる。性能リスクを未然に防ぐ判断。上限超過時はエラーメッセージで明示する。

### なぜ他の案を採らなかったか（必須）

- 案 2（上限なし）: 大量データで OOM / タイムアウトが発生するリスクを DB タイムアウトに委ねることになり、エラーが予測不能なタイミングで起きる。既存バッチとの上限不整合も生まれる。
```

**例 2: 種別「妥協」**

```markdown
# DR-012: alertReason 重複排除を一旦アプリ側で実装

- **Phase**: T3（実装）
- **日時**: 2026-05-20T15:18:00+09:00
- **種別**: 妥協
- **Spec参照**: APP-1561 AC5（重複した alertReason は 1 件として扱う）
- **可逆性**: 中（DB 側に UNIQUE 制約を追加すれば移行できるが既存データの cleanup が必要）
- **フォローアップ**: APP-1700 で DB 制約化を計画
- **状況**: 理想は DB レベルの UNIQUE 制約だが、既存データに重複が含まれており、制約追加には事前 cleanup が必要。本 PR スコープ外。

## 選択肢

| # | 選択肢 | メリット | デメリット |
|---|--------|---------|----------|
| 1 | アプリ側で distinct 取得 | 即対応可能、PR スコープ内 | 重複自体は DB に残る、他経路から書き込まれると再発 |
| 2 | DB に UNIQUE 制約 + cleanup マイグレーション | 根本対策 | スコープ外、影響範囲が広い |

## 決定

**選択: 1** — アプリ側 distinct

## 理由

スコープを守るため理想解を諦めた妥協。フォローアップとして APP-1700 で DB 制約化を計画し、それまでは他経路からの書き込みを抑止する責任は本 UseCase にあると明記。

### なぜ他の案を採らなかったか（必須）

- 案 2（DB UNIQUE 制約 + cleanup マイグレーション）: 根本対策だが、既存データの重複 cleanup が必要で本 PR スコープ外。影響範囲が広く、誤った cleanup で既存データを破壊するリスクもある。APP-1700 で改めて対応する。
```

### trivial 除外基準（DR 不要）

以下に該当する変更は DR を起こさない（過剰生成を防ぐ）。

- rename / import 追加 / typo 修正
- 既存パターンを単純踏襲しただけの実装
- 1 行未満の変更、または機械的な置換のみ

### ステージあたり推奨上限

spec 外種別（明示外 / 妥協 / 先送り / スコープ追加）が **1 ステージで 3 件を超えたら**、それ自体が設計の問題シグナル。E3（チケット再分割）への差し戻しを検討する。

# 入力判定

`$ARGUMENTS` にチケットID・URL・または作業内容テキストを渡す。**2軸**で判定する。

### 軸1: チケットタイプ（フロー選択）

| チケットタイプ | 判定基準 | アクション |
|---------------|---------|-----------|
| **Epic** | issue_type が Epic、または子チケット/サブタスクを持つ | **上位フロー E1 へ** |
| **User Story** | issue_type が Story/US、またはACが複数の独立機能を含む | **上位フロー E1 へ** |
| **実装チケット** | issue_type が Task/Sub-task、またはスコープが単一機能 | **下位フロー T1 へ** |
| **判定不能** | Decision Recordに記録し、安全側（上位フロー）で進む | **上位フロー E1 へ** |

> **⚠️ 下位フローの実行ルール（最重要）**
>
> 下位フロー T1〜T5 は**一気通貫で実行する**。T3（実装）完了後にユーザーに確認を求めてはならない。
> - T3完了 → **即座にQG（品質ゲート）を起動** → QG完了 → T5（Push確認）
> - 途中で「PR作りますか？」「次に進みますか？」と聞くのは**フロー違反**

### 軸2: 作業モード（新規 or 修正）

チケット内容・AC・関連コードから作業モードを判定し、フロー全体に引き継ぐ。

| 作業モード | 判定基準 | フローへの影響 |
|-----------|---------|--------------|
| **新規開発** | 対象機能がまだ存在しない。新しいエンドポイント/画面/ドメインの追加 | 通常のフロー実行 |
| **既存修正** | 対象機能のコードが既に存在する。バグ修正、仕様変更、リファクタリング、機能拡張 | 各ステップで**修正時の目的**に従って実行（下記参照） |

> **⚠️ アンチパターン: 「既に実装済みだからスキップ」**
>
> 修正タスクで以下の判断をしてはならない:
> - 「コードが既にある → E2（設計）は不要」→ **誤り。修正の影響波及を調査するためにE2は必須**
> - 「テストが既にある → 検証基準定義は不要」→ **誤り。修正後の期待動作を定義するためにT3内の検証基準定義は必須**
> - 「小さな変更 → QG（品質ゲート）を簡略化」→ **誤り。小さな変更こそ既存コードへの副作用リスクがある**
>
> **修正時こそフローの各ステップが重要になる。** 各ステップの修正時の具体的な目的は upper-flow.md / lower-flow.md に記載。

# フロー詳細（参照ファイル）

各フローの詳細手順は参照ファイルに分離されている。該当するフローに進む際に読み込むこと。

## 要求定義（E1 前の任意フェーズ）

Epic/US の要求が曖昧・未整理な場合、E1 の前に要求整理を行い、仕様を明確化してからフローに入る。要求が十分に明確な場合はスキップしてよい。

| 状況 | 使用スキル |
|------|-----------|
| 対話形式で要求を引き出したい（要求が不明確・口頭段階） | **`software-requirements`** スキルを呼び出す |
| チケットを渡してフォーマルな設計書を生成したい | **`sier-dev`** スキルを呼び出す（要件定義書〜内部設計書まで生成可） |

`sier-dev` で生成した `docs/01_requirements/要件定義書.md` は E1 の仕様書ソースとして扱う。

## 上位フロー（E1〜E4）

**→ `references/upper-flow.md` を読み込む**

Epic/USチケットから設計→チケット分割→実行ループまでの全手順。E&C（Expand and Contract）のチケット分割ルール・テンプレート・具体例も含む。旧E4（AC生成）はE1に統合、旧E5はE4に繰り上げ。

## 下位フロー（T1〜T5）

**→ `references/lower-flow.md` を読み込む**

個別の実装チケットに対するWorktree作成→実装→品質ゲート→PR作成の全手順。旧T3-preはT3に統合、旧T3b+T4+T5は「QG（品質ゲート）」サブエージェントに統合。Team / file-handoff ワーカーモード、SendMessage / events.log 通信プロトコルも含む。

> **フロー実行の鉄則**: T3完了後、QGを経由せずにT5（PR作成）に進んではならない。T3→QG→T5は一気通貫。

## 最終報告書

**→ `references/final-report.md` を読み込む**

作業完了時に生成する最終報告書のテンプレートと生成ルール。上位フロー（E4完了後）・下位フロー単独（T5 Push確認前・PR作成前）の両方で必ず実行する。

## 横断的ルール（全フェーズ共通）

**→ `references/quality-rules.md` を読み込む**

Verification Before Completion 鉄則と Anti-rationalization パターン集。フロー実行開始時に読み込み、全フェーズで適用する。

## ドキュメント同期（SSOT）

**→ `references/doc-sync.md` を読み込む**

各フェーズ完了時に「今どの情報を根拠に作業しているか」を `scratch.md` に宣言するプロトコル。E1完了→`spec-draft.md`、E2完了→`design.md`、E3完了→`ticket-plan.md` の順にSSOTが移行する。

## サブエージェント通信プロトコル

**→ `references/subagent-protocol.md` を読み込む**

ワーカー↔Team Lead 間のステータス定義と処理フロー。E4（Team / file-handoff オーケストレーション）で使用する。

## 影響範囲調査（E2a 必須読み込み）

**→ `references/impact-analysis.md` を読み込む**

E2a（影響範囲調査）で 3 並列 `Explore` に渡す **7観点チェックリスト** と影響範囲レポートの出力フォーマット。
「3 並列で見ろ」だけでは観点抜けが発生し、後段の QG / レビューで IDOR / fail-open /
副作用見落とし / 既存テスト破壊などの形で噴出する。E2a 着手時に必ず読み込む。

`refine-ticket` も同じ 7観点フレームを共有する（軽量版）。E1 で `refine-ticket` が生成した
軽量影響範囲レポート（spec-draft.md）を出発点として、E2a で設計レベルの完全版に拡張する。
