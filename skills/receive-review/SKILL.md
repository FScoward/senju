# レビュー対応スキル（receive-review）

PRのレビュー指摘を受け取り、「何を・どう直すか」を素早く整理し、次回への学びまでつなげる。

## なぜこのスキルが必要か

レビュー指摘を受けた時の課題：
- 「全部対応すべき？」の判断コスト
- 「どこから手をつけよう」の優先度迷子
- 指摘が多いと対応漏れが発生しやすい
- 同じ指摘を繰り返し受けてしまう

このスキルはこれらを一気に解決する。

---

## Step 1: 指摘内容の収集

ユーザーからの入力を受け取る。

**PR URL の場合:**
```bash
gh pr view <PR番号> --comments --json body,comments -q '
  "PR本文: " + .body + "\n\n" +
  (.comments | map("[@" + .author.login + "]: " + .body) | join("\n\n"))
'
```

**インラインコメントも含める場合:**
```bash
gh api repos/<owner>/<repo>/pulls/<PR番号>/comments \
  --jq '.[] | "[\(.path):\(.line)] @\(.user.login): \(.body)"'
```

**テキスト貼り付けの場合:** ユーザーが貼ったテキストをそのまま処理する。

---

## Step 2: 指摘の分類

各指摘を以下の4カテゴリに分類し、理由を添える。

| カテゴリ | 記号 | 判断基準 |
|---------|------|---------|
| **MUST** | 🔴 | バグ・セキュリティ・要件違反・マージブロッカー・データ整合性に影響 |
| **SHOULD** | 🟡 | コード品質・保守性・テスト不足・コーディング規約違反・既存パターンとの不整合 |
| **NICE** | 🟢 | スタイル・好みの問題・微妙な改善提案・パフォーマンス最適化（影響軽微） |
| **DISCUSS** | 💬 | 設計方針・要件の解釈・実装の選択肢が複数ある・レビュアーの意図が不明確 |

分類に迷う時の判断軸：
- **本番でバグが起きうるか？** → MUST
- **次のエンジニアが困るか？** → SHOULD
- **気になるけど動作に影響しない？** → NICE
- **答えが1つでなく議論が必要？** → DISCUSS

### 出力フォーマット

```
## レビュー指摘トリアージ結果

### 🔴 MUST（必須対応: N件）
1. [ファイル名:行番号 または コメントID] 指摘の要約（1行）
   **指摘内容**: レビュアーのコメント原文（要約）
   **分類理由**: なぜMUSTか
   **対応方針**: 具体的にどう直すか

### 🟡 SHOULD（推奨対応: N件）
（同形式）

### 🟢 NICE（任意対応: N件）
（指摘要約のみ。対応するかどうかはあなたの判断で）

### 💬 DISCUSS（要議論: N件）
（指摘要約 + 「何について議論が必要か」）
```

---

## Step 3: 対応方針の立案（MUST / SHOULD）

MUST・SHOULDの各指摘について具体化する。

**各指摘に対して以下を明確にする:**

1. **変更対象**: ファイル名・関数名・変更内容の概要
2. **変更方針**: どう直すか（複数案がある場合は推奨案と理由を添える）
3. **依存関係**: 他の指摘と連動して直すべきものはあるか
4. **所要感**: 小（5分以内）/ 中（30分以内）/ 大（それ以上）

**5件以上の場合は関連指摘をグルーピングする** — 同じファイル・同じ設計上の問題に起因する指摘はまとめて対応すると効率的。

---

## Step 4: TodoWriteでタスク化

MUST → SHOULD の順でタスクリストを作成する。

- タスク名: `[ファイル名] 変更内容の一行サマリ`
- MUST は全件タスク化
- SHOULD は対応方針が固まったものをタスク化
- NICE・DISCUSS はタスク化しない（後で判断するため）

---

## Step 5: 再発防止ルール候補の抽出

今回の指摘パターンから「次回同じ指摘を受けないためのルール」をゼロベースで考え、**提案として示す**。

LLMへの指示ファイルへの追記は行わない。LLMに依存したルール適用は非決定論的であり、静的に保証できないため。再発防止の本命は、コードやCI/CD上に根付く仕組みにある。

### ルール候補の判断基準

以下に当てはまる指摘はルール化の価値が高い:
- 「言われてみれば確かに」と納得感がある（再現性が高い）
- コードレビューで繰り返し発生しそうなパターン
- 実装前に意識していれば避けられたもの

逆に以下はルール化しない:
- 今回のPR固有の文脈による指摘（仕様・要件の問題）
- 「好みの問題」レベルのもの（個人差がある）

### ルール候補の出力フォーマット

各候補に対して「どの仕組みで防げるか」と「手軽さ」を示す。実際に導入するかはユーザーの判断に委ねる。

```
## 📌 再発防止ルール候補

### ルールタイトル
- **問題**: なぜこの指摘が来たか（1〜2文）
- **NG例**: （コードスニペット）
- **OK例**: （コードスニペット）
- **防止手段の候補**（効果が高い順）:
  - 🔧 lint: ktlint / ESLint カスタムルール化（静的保証・CI強制）
  - 🧪 arch test: ArchUnit でアーキテクチャ制約をテスト化（静的保証）
  - 📋 PR template: PR テンプレートのチェックリストに追加（人間チェック）
  - 📄 review-rules: `~/.claude/review-rules/` に追記（次回レビュー時のヒント）
```

`~/.claude/review-rules/` への追記をユーザーが選んだ場合のみ実施する（自動追記はしない）。
lint や arch test 化できるルールは、そちらを優先して提案する。

---

## Step 6: 対応サマリの出力

```
## 対応サマリ

- 🔴 MUST: X件 → 全て対応必須
- 🟡 SHOULD: X件 → 対応推奨
- 🟢 NICE: X件 → 任意（今回はスキップ推奨）
- 💬 DISCUSS: X件 → レビュアーに確認が必要

📝 タスク作成: X件
📌 再発防止ルール候補: X件（承認待ち）

次のアクション:
1. TodoリストのMUSTタスクから着手
2. DISCUSSはPRコメントで返信・確認
3. ルール候補を確認して追記するか判断
```

---

## 補足: DISCUSS の返信テンプレート

DISCUSSに分類された指摘には、PRコメントで返信が必要。以下を参考に返信案を提案する。

**意図確認が必要な場合:**
```
ご指摘ありがとうございます。
〇〇の方針についてご意見いただきたいのですが、
A案（現状の実装）とB案（〇〇に変更）のどちらがよいでしょうか？
```

**方針を説明して確認する場合:**
```
〇〇については〇〇の理由でこの実装にしました。
ご懸念の点は〇〇でしょうか？別のアプローチのご提案があれば教えていただけますか？
```
