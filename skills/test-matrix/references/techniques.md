# テスト設計技法リファレンス

各技法の「やること」「導出手順」「実例」を記載する。

---

## 1. 同値分割（Equivalence Partitioning）

**考え方**: 同じ振る舞いをするはずの入力をグループ（等価クラス）にまとめ、
各グループから代表値を1つ選ぶ。グループ内はどれを選んでも同じ結果なので、
全パターンを試す必要はない。

**導出手順**:
1. 入力の有効範囲・無効範囲を列挙する
2. 各範囲が1つの等価クラス
3. クラスごとに代表値を1つ選ぶ（中間値が一般的）

**例: 年齢入力（18〜65歳が有効）**

| クラス | 範囲 | 代表値 | 種別 |
|---|---|---|---|
| 有効クラス1 | 18〜65 | 30 | 正常系 |
| 無効クラス1 | 65より大 | 100 | 異常系 |
| 無効クラス2 | 18未満 | 10 | 異常系 |
| 無効クラス3 | 非整数 | "abc" | 異常系 |
| 無効クラス4 | null/空 | null | 異常系 |

## 2. 境界値分析（Boundary Value Analysis）

**考え方**: バグは境界付近に集中する（off-by-one エラーなど）。
等価クラスの境界値とその前後を必ずテストする。

**導出手順**:
1. 境界を特定する（min, max, 閾値など）
2. 境界値・境界-1・境界+1 の3点をテストする
3. 両端（最小・最大）両方に適用する

**例: 年齢入力（18〜65歳が有効）**

| テスト値 | 説明 | 期待結果 |
|---|---|---|
| 17 | 最小値-1 | エラー |
| 18 | 最小値（下境界） | 成功 |
| 19 | 最小値+1 | 成功 |
| 64 | 最大値-1 | 成功 |
| 65 | 最大値（上境界） | 成功 |
| 66 | 最大値+1 | エラー |

**よくある境界**:
- 数値: 0, -1, 1, Integer.MAX_VALUE, Integer.MIN_VALUE
- 文字列: 空文字(""), 1文字, 最大長, 最大長+1
- 配列/リスト: 0件, 1件, N件, N+1件
- null / undefined / 空

## 3. デシジョンテーブル（Decision Table Testing）

**考え方**: 複数の条件（True/False）の組み合わせと、それに対するアクションを表で整理する。
条件が3つなら最大 2³=8 パターン。論理的に同じ結果になるパターンを統合して削減できる。

**導出手順**:
1. 条件（入力・フラグ・状態）を列挙する
2. 全組み合わせを作る（2^n パターン）
3. 各組み合わせの期待アクション（出力・副作用）を決める
4. 同じアクションになる列はまとめて削減する

**例: ローン審査（年収・勤続年数・信用スコアで判定）**

| 条件 | C1 | C2 | C3 | C4 | C5 |
|---|---|---|---|---|---|
| 年収 >= 400万 | T | T | T | F | F |
| 勤続年数 >= 3年 | T | T | F | T | F |
| 信用スコア >= 700 | T | F | - | - | - |
| **アクション** | **承認** | **条件付承認** | **否決** | **条件付承認** | **否決** |

`-` は「どちらでも結果が同じ」。

## 4. 状態遷移テスト（State Transition Testing）

**考え方**: オブジェクトやシステムが持つ「状態」と「状態を変えるイベント」を図にし、
すべての遷移パスをカバーする。

**導出手順**:
1. 状態を列挙する（例: DRAFT, IN_REVIEW, APPROVED, REJECTED）
2. イベント（遷移トリガー）を列挙する
3. 状態×イベントの組み合わせで「正常遷移」と「無効遷移」を特定する
4. 各遷移をテストケースにする

**テスト種別**:
- **0スイッチカバレッジ**: すべての状態を少なくとも1回通る
- **1スイッチカバレッジ**: すべての状態遷移（辺）を少なくとも1回通る（基本はこれ）
- **2スイッチカバレッジ**: 遷移の連続ペアをすべてカバー（より厳密）

**例: タスクの状態遷移**

```
[TODO] --開始--> [IN_PROGRESS] --完了--> [DONE]
                      |
                    差し戻し
                      ↓
                   [REJECTED] --再開--> [IN_PROGRESS]
```

| TC | 開始状態 | イベント | 期待終了状態 | 分類 |
|---|---|---|---|---|
| TC-1 | TODO | 開始 | IN_PROGRESS | 正常遷移 |
| TC-2 | IN_PROGRESS | 完了 | DONE | 正常遷移 |
| TC-3 | IN_PROGRESS | 差し戻し | REJECTED | 正常遷移 |
| TC-4 | REJECTED | 再開 | IN_PROGRESS | 正常遷移 |
| TC-5 | TODO | 完了 | - | 無効遷移（エラーになるべき） |
| TC-6 | DONE | 開始 | - | 無効遷移（エラーになるべき） |

## 5. ペアワイズテスト（Pairwise Testing / All-Pairs）

**考え方**: 独立した複数のパラメータがあるとき、全組み合わせは爆発するが、
「どの2つのパラメータの組み合わせも少なくとも1回カバー」することで
バグの多くを検出できる。組み合わせ数を大幅に削減できる。

**導出手順**:
1. パラメータと各パラメータの取りうる値を列挙する
2. ペアワイズツールか手動でカバー表を作る
3. すべての2パラメータペアが全値の組み合わせで登場するか確認する

**例: 検索フィルター（OS × ブラウザ × 言語）**

- OS: Windows, Mac, Linux
- ブラウザ: Chrome, Firefox, Safari
- 言語: ja, en

全組み合わせ: 3×3×2 = 18件
ペアワイズ後: 約9件に削減

| TC | OS | ブラウザ | 言語 |
|---|---|---|---|
| 1 | Windows | Chrome | ja |
| 2 | Windows | Firefox | en |
| 3 | Windows | Safari | ja |
| 4 | Mac | Chrome | en |
| 5 | Mac | Firefox | ja |
| 6 | Mac | Safari | en |
| 7 | Linux | Chrome | ja |
| 8 | Linux | Firefox | en |
| 9 | Linux | Safari | ja |

## 6. ユースケーステスト（Use Case Testing）

**考え方**: ユーザーが実際に行う操作フロー（シナリオ）を通してシステムを検証する。
単体の入力ではなく「一連の操作」を対象にする。

**構成要素**:
- **主シナリオ（成功シナリオ）**: ユーザーが期待通りゴールに到達するフロー
- **代替シナリオ**: 別の正常な経路
- **例外シナリオ**: エラーや失敗から回復するフロー

**例: ユーザー登録フロー**

```
主シナリオ:
1. メールアドレスを入力
2. パスワードを入力（8文字以上）
3. 確認メールを受信
4. リンクをクリックして認証完了

例外シナリオ1: メールアドレスが既に登録済み → エラーメッセージ表示
例外シナリオ2: パスワードが弱すぎる → インライン警告
例外シナリオ3: 確認メールの期限切れ → 再送ボタン表示
```

## 7. エラー推測（Error Guessing）

**考え方**: テスト設計者の経験則から「ここが壊れやすい」を直感的に洗い出す。
他の技法でカバーされない「ありがちなバグ」を狙い撃ちする。

**よくあるエラー推測パターン**:

### 入力値系
- `null` / `undefined` / 空文字 / 空配列 / 空オブジェクト
- ゼロ除算（分母が0になるケース）
- 整数オーバーフロー（Long.MAX_VALUE + 1 など）
- 特殊文字: `'` `"` `<` `>` `&` `\n` `\0` （SQLインジェクション・XSS系）
- Unicode: 絵文字・全角スペース・制御文字・サロゲートペア
- 先頭・末尾の空白（トリミング漏れ）

### 数値・演算系
- 浮動小数点誤差: `0.1 + 0.2 == 0.3` → `false`
- 丸め誤差・切り捨て vs 四捨五入
- 負の値・マイナスの時間差

### 状態・タイミング系
- 同一リソースへの同時アクセス（競合状態）
- トランザクション途中でのエラー（ロールバックされるか）
- リトライ時の冪等性（2回実行しても同じ結果になるか）
- キャッシュが古い状態で読まれるケース

### 権限・認証系
- 他テナントのリソースへのアクセス（IDOR）
- ログアウト後のセッション再利用
- 権限が変わった直後のキャッシュ利用

### 外部依存系
- 外部API がタイムアウト・500を返す
- ネットワーク断切（途中でリクエストが途切れる）
- DBが遅延してタイムアウト

## 8. リスクベーステスト（Risk-Based Testing）

**考え方**: すべてのテストを均等に扱わず、「失敗したときの影響度」×「失敗の発生確率」で
優先順位をつける。時間が限られているときに何を先にテストすべきかを決める。

**リスクマトリクス**:

```
発生確率
  高 │ 中優先 │ 最高優先 │
     │────────│──────────│
  低 │ 低優先 │ 高優先   │
     └────────────────────
        低影響   高影響
                        影響度
```

**優先度を高くすべきケース**:
- 金融処理・決済フロー（影響度: 高）
- 認証・認可（セキュリティリスク: 高）
- データ移行・バッチ処理（復元困難: 高）
- ユーザーが最も使う主要フロー（使用頻度: 高）

**優先度を低くできるケース**:
- 管理者専用の稀な操作
- UIの細かな見た目の調整
- ログ出力の詳細フォーマット

## 技法選択のチートシート

```
入力パラメータがある
  → 同値分割（必須）
  → 境界値分析（数値・文字数・配列長があれば必須）

複数の条件フラグ・設定の組み合わせがある
  → デシジョンテーブル（条件が2〜4個）
  → ペアワイズ（パラメータが5個以上）

状態を持つオブジェクト・ワークフローがある
  → 状態遷移テスト

ユーザー操作フローをE2Eで検証したい
  → ユースケーステスト

「何が壊れやすいか」直感で洗い出したい
  → エラー推測

テスト優先順位を決めたい
  → リスクベーステスト
```
