# test-matrix

仕様・コード・関数・API を受け取り、**テスト設計技法を組み合わせた網羅的なテストマトリクス**を生成する。

---

## なぜテスト設計技法が必要か

「思いつくままにテストを書く」と、同じ正常系ばかり増えて境界やエラー系が抜ける。
テスト設計技法は「どこを見落としやすいか」の経験則を体系化したもの。
技法ごとに視点が違うため、組み合わせることで死角を埋められる。

---

## ステップ

### Step 1: 対象を把握する

ユーザーが渡したもの（コード・仕様文・API定義・関数シグネチャ）を読み、以下を特定する：

- **入力**: パラメータ・フォームフィールド・リクエストボディなど
- **条件分岐**: if/when/switch・バリデーションルール・権限チェック
- **状態**: オブジェクトやDBが取りうる状態、状態を変えるイベント
- **出力**: 戻り値・レスポンス・副作用（DB書き込み・メール送信など）

### Step 2: 適用する技法を選ぶ

対象の特性に合わせて技法を選択する。複数を組み合わせるほど網羅性が上がる。

| 技法 | いつ使う |
|---|---|
| 同値分割 | 入力に「有効な範囲」と「無効な範囲」がある |
| 境界値分析 | 数値・文字数・配列長など境界がある |
| デシジョンテーブル | 複数条件の組み合わせで結果が変わる |
| 状態遷移 | オブジェクト・ワークフローに状態がある |
| ペアワイズ | 独立した複数パラメータの組み合わせ爆発を抑えたい |
| ユースケース | ユーザーの操作フローをシナリオとして検証したい |
| エラー推測 | 経験則から「ここが壊れやすい」を洗い出したい |
| リスクベース | テストの優先順位を付けたい |

**技法の自動適用ルール（これらに該当したら必ず使う）**:

- 条件フラグ・設定が 2〜4 個あり AND/OR で組み合わさる → **デシジョンテーブル必須**（2^n 全パターンを表にする）
- パラメータが 5 個以上で全組み合わせが爆発する → **ペアワイズで削減**し、なぜ削減したか説明する
- 数値・文字数・配列長の境界がある → **境界値分析必須**（境界-1, 境界, 境界+1 の3点）
- 状態（enum/ステータス）を持つ対象 → **状態遷移テスト必須**（0スイッチ：全状態を通る、1スイッチ：全遷移を通る、無効遷移もカバー）
- 入力パラメータが存在する → **同値分割は常に適用**（null/空/不正値クラスを忘れない）

### Step 3: 各技法でテストケースを生成する

詳細な技法ガイドは `references/techniques.md` を参照。

技法ごとに以下を導出する：
1. **等価クラス / 条件セット / 状態セット** を列挙
2. 各クラスから **代表値** を選ぶ
3. **正常系・異常系・境界値** をそれぞれカバーしているか確認

**状態遷移テストを適用するとき**は必ず以下の2段階でカバーする：
- **0スイッチカバレッジ**: 全状態を少なくとも1回通る（最低限）
- **1スイッチカバレッジ**: 定義された全遷移を少なくとも1回通る（標準）
- **無効遷移**: 定義されていない状態×イベントの組み合わせがエラーになることを確認する
- **状態×イベントマトリクス**（行=現在状態、列=イベント、セル=✅遷移先 or ❌エラー）を必ず添付する

### Step 4: テストマトリクスを出力する

**出力形式の絶対ルール**: テストケースは必ず `|` 区切りのマークダウン表形式で出力する。箇条書き・`###` 見出し＋説明文・インデント形式は使わない。各行が1テストケース、各列が属性になるよう統一する。

以下の形式でマトリクスを生成する：

```markdown
## テストマトリクス: {対象の名前}

### 前提・仕様サマリ
{分析した入力・条件・状態・出力を箇条書きで}

### テストケース一覧

| ID | 技法 | 分類 | テスト観点 | 入力 / 操作 | 期待結果 | 優先度 |
|---|---|---|---|---|---|---|
| TC-001 | 同値分割 | 正常系 | 有効な年齢 | age=25 | 登録成功 | 高 |
| TC-002 | 境界値 | 境界 | 最小年齢 | age=18 | 登録成功 | 高 |
| TC-003 | 境界値 | 境界 | 最小年齢-1 | age=17 | バリデーションエラー | 高 |
...

### カバレッジサマリ

- 正常系: N件
- 異常系: N件  
- 境界値: N件
- 状態遷移: N件（該当する場合）
- 組み合わせ（ペアワイズ）: N件（該当する場合）

### 未カバー・検討事項
{カバーできなかった観点や、実装を見ないと判断できない点}
```

---

## 優先度の目安

| 優先度 | 基準 |
|---|---|
| 高 | 正常系メインパス・境界値・セキュリティ境界 |
| 中 | 異常系・準正常系・組み合わせ |
| 低 | エラー推測・探索的・nice-to-have |

---

## 他スキルとの連携

- **TDD前**: テストマトリクスを先に出してから Red フェーズに入ると、何を実装すべきかが明確になる
- **レビュー後**: `review-loop` や `review-loop-duo` の指摘でテスト漏れが判明したら、このスキルで補完する
- **チケット起票**: `refine-ticket` と組み合わせてテスト観点をACに落とし込む
